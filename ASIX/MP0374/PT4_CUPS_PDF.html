Context de la pràctica

Per a quin motiu és rellevant aquesta pràctica?

En moltes empreses coexisteixen equips Windows i Linux. Gestionar la impressió de forma centralitzada, garantir que cada usuari accedeix únicament als seus documents i limitar el consum de recursos (quotes) és una tasca real d'un administrador de sistemes.

💡

Idees clau d'aquesta pràctica

  • No hi ha impressora física: CUPS-PDF genera documents PDF com a resultat de cada "impressió".
  • Tot es controla des del servidor: els usuaris imprimeixen des del client, però el PDF es genera i s'emmagatzema al servidor Zentyal.
  • Permisos segregats: cada grup (alumnes / professors) accedeix únicament als seus documents PDF i a la seva impressora virtual.
  • Quotes de documents: un script Bash limita el nombre de PDFs que cada usuari pot tenir emmagatzemats simultàniament.

Conceptes teòrics previs

El que has de comprendre abans de començar

Aquesta pràctica combina tres tecnologies: CUPS (servidor d'impressió), SMB/Samba (compartició de fitxers en xarxes heterogènies) i Bash scripting (automatització). Has d'entendre cadascuna per poder configurar-les correctament.

CUPS: Common Unix Printing System

Arquitectura de CUPS

CUPS és el sistema d'impressió estàndard de Linux i macOS. Funciona com un servidor d'impressió: les aplicacions envien documents a CUPS, que els processa i els envia a la impressora corresponent. Es gestiona amb una interfície web a http://localhost:631. Permet compartir impressores en xarxa amb protocols estàndard (IPP, Samba).

CUPS-PDF: la impressora virtual

CUPS-PDF és un backend de CUPS que, en lloc d'enviar el document a una impressora física, el converteix en un fitxer PDF i el desa a una ruta del sistema. És la solució estàndard per a entorns sense impressora física o per a arxivar documents en format PDF des de qualsevol aplicació.

Cas d'ús real

Una empresa volia que els comercials "imprimissin" els albarans directament en PDF al servidor, sense usar paper. CUPS-PDF va ser la solució.

El directori de sortida i el PostProcessing

CUPS-PDF guarda el PDF a una ruta temporal. La directiva Out del fitxer cups-pdf.conf defineix on va el PDF. La directiva PostProcessing permet executar un script just després de generar el PDF: és l'eina que farem servir per moure cada PDF al directori correcte de cada usuari/grup.

Variables d'entorn de CUPS-PDF

Quan CUPS-PDF executa el script de post-processat, li passa informació via variables d'entorn: $USER (l'usuari que ha imprès), $PRINTER (la impressora usada), $FILE (ruta al PDF temporal). El script ha de llegir i usar aquestes variables per prendre les decisions correctes.

SMB/Samba: compartició en xarxes heterogènies

Per a quin motiu SMB?

SMB (Server Message Block) és el protocol que usa Windows per compartir fitxers i impressores en xarxa. Linux implementa SMB a través de Samba. Gràcies a Samba, un servidor Linux (com Zentyal) pot actuar com un servidor de fitxers i impressores completament transparent per a clients Windows, sense que l'usuari noti la diferència.

Accés des de Windows vs. Linux

Des de Windows, s'accedeix als recursos compartits via l'explorador d'arxius: \\IP_SERVIDOR\recurs. Des de Linux, s'usa el muntatge SMB o l'accés via gestor de fitxers: smb://IP_SERVIDOR/recurs. En ambdós casos, el servidor Samba valida les credencials de l'usuari i aplica els permisos configurats.

Permisos POSIX i control d'accés

Permisos 700: carpeta privada

El permís 700 significa: el propietari té lectura+escriptura+execució; el grup i la resta no tenen cap permís. Per a directoris d'usuari, és el permís adequat per garantir que ningú més pot accedir als documents.

Notació octal

7 = rwx (propietari)
0 = --- (grup)
0 = --- (resta)

Propietari i grup del fitxer

Cada fitxer i directori de Linux té un propietari (un usuari) i un grup associat. El script de post-processat ha d'assignar el propietari correcte a cada PDF generat (chown usuari:usuari fitxer.pdf) perquè l'usuari pugui llegir-lo a través de la compartició SMB.

Quotes de documents

Una quota limita el nombre de recursos (en aquest cas, PDFs) que un usuari pot tenir. En aquesta pràctica, les quotes no es gestionen amb eines del kernel (com quota) sinó amb un script Bash que compta els PDFs existents i refusa la nova impressió si se supera el límit.

Bash Scripting: automatització al servidor

Per a quin motiu un script i no una solució manual?

Sense script, el PDF quedaria en un directori temporal i ningú el podria trobar. El script de post-processat de CUPS-PDF és el cervell del sistema: llegeix variables d'entorn, decideix on mou el PDF (alumnes o professors), crea el directori si no existeix, aplica permisos i registra l'operació al log.

Conceptes de Bash que aplicaràs

El professor t'explicarà el codi del script, però és important que entenguis: variables ($VAR), condicionals (case/if), comandes de fitxers (mkdir -p, mv, chmod, chown), redireccions (>> log) i substitució de comandes ($(comanda)). Cada element del script té un propòsit concret.

Tasques d'ampliació

Control de quotes d'impressió

Les tasques d'ampliació implementen un sistema de quotes que limita el nombre de PDFs que cada usuari pot acumular. Un script Bash comprova el comptador i refusa la impressió si s'ha superat el límit.

ℹ️

Lògica del sistema de quotes

El sistema de quotes funciona de la manera següent: els alumnes tenen un límit de 5 PDFs i els professors de 10 PDFs. Quan un usuari intenta imprimir i ja ha superat el límit, el PDF nou es descarta i s'escriu un fitxer ATENCIO_QUOTA.txt al seu directori explicant la situació i com resoldre-la.

Pensa en: flexibilitat, rendiment, atomicitat, condicions de competència (race conditions), complexitat d'implementació.
