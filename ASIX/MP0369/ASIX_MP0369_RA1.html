1

Estructura d'un sistema informàtic

Hardware, software, firmware i la seva interrelació. La base sobre la qual s'executa tot sistema operatiu.

Fonaments

Capes d'un sistema informàtic

Un sistema informàtic és el conjunt format per hardware (components físics), firmware (capa intermèdia persistent) i software (programes). Entendre aquesta estructura en capes és imprescindible per comprendre com un SO actua com a intermediari entre el hardware i les aplicacions.

🖥️

Hardware (Maquinari)

Tots els components físics del sistema. Classifiquem en:

  • ⚙️ Processador (CPU): executa instruccions. Arquitectures x86-64, ARM, RISC-V.
  • 🧠 Memòria RAM: emmagatzematge volàtil. El SO hi carrega els processos actius.
  • 💾 Emmagatzematge: HDD, SSD (SATA, NVMe). Persistent. Conté el SO i les dades.
  • 🔌 Placa base: interconnecta tots els components. Conté el xip del firmware.
  • 📡 Perifèrics: d'entrada (teclat, ratolí), de sortida (monitor, impressora) i mixtos (NIC, USB).
⚡

Firmware

Software de baix nivell emmagatzemat en memòria no volàtil (ROM, Flash EEPROM) integrada al hardware. No es perd en apagar l'equip.

  • 📟 BIOS/UEFI: firmware de la placa base. Primera cosa que s'executa en encendre l'equip.
  • 🖨️ Firmware de perifèrics: discs durs, impressores, targetes de xarxa, routers.
  • 📱 Bootloader: programa petit que carrega el SO a la RAM (tècnicament és firmware del SO).
⚠️

Actualitzar el firmware d'una placa base incorrectament pot inutilitzar (brick) l'equip permanentment. Cal molta precaució.

💿

Software (Programari)

Conjunt d'instruccions lògiques. Es classifica en tres grans capes:

  • 🏗️ Software de sistema: el Sistema Operatiu i les seves eines (drivers, utilitats).
  • 🔧 Software de suport (middleware): SGBD, servidors web, runtimes (JVM, .NET).
  • 📱 Software d'aplicació: programes que usa l'usuari final (ofimàtica, navegadors, jocs).
🔑
Relació entre capes — el SO com a intermediari:

El Sistema Operatiu actua com a interfície entre el hardware i el software d'aplicació. Cap aplicació accedeix directament al hardware: ho fa a través de les crides al sistema (system calls) que el SO exposa. Això garanteix seguretat, aïllament i portabilitat.

🔑 Components del Kernel

  • ⚙️ Planificador de processos (scheduler): decideix quin procés s'executa en cada moment. Algoritmes: Round Robin, CFS (Linux), MLFQ (Windows).
  • 🧠 Gestor de memòria: assigna i allibera RAM. Implementa la memòria virtual (paginació, segmentació).
  • 📁 Sistema de fitxers virtual (VFS): abstracció unificada sobre NTFS, ext4, FAT32, etc.
  • 🔌 Gestió d'E/S i IRQ: interruptes hardware, DMA, drivers.
  • 🌐 Pila de xarxa: TCP/IP, protocols de xarxa integrats al kernel.
Tema 3

Funcions d'un sistema operatiu

El SO realitza cinc grans funcions que permeten als programes executar-se de manera eficient, segura i ordenada sobre el hardware.

⚙️

1. Gestió de processos

Crea, planifica, suspèn i elimina processos. Gestiona la concurrència (múltiples processos en aparent simultaneïtat) i la comunicació entre processos (IPC: pipes, sockets, memòria compartida). En Windows: Administrador de Tasques. En Linux: ps, top, htop.

🧠

2. Gestió de memòria

Assigna espais de memòria als processos. Implementa memòria virtual (cada procés creu tenir memòria contigua) usant paginació. Gestiona el swap (àrea d'intercanvi en disc quan la RAM s'omple). Protegeix que un procés no accedeixi a la memòria d'un altre.

📁

3. Gestió del sistema de fitxers

Organitza les dades en fitxers i directoris. Gestiona els permisos d'accés (lectura, escriptura, execució). Suporta múltiples sistemes de fitxers: NTFS (Windows), ext4/XFS/Btrfs (Linux), FAT32 (compatible universal), exFAT (USB/targetes).

🔌

4. Gestió de dispositius (I/O)

Gestiona la comunicació amb el hardware a través dels controladors (drivers). Proporciona una interfície uniforme: una aplicació llegeix un fitxer igual si és en un SSD, un NAS o un CD. Inclou la gestió d'interrupcions (IRQ) i DMA.

🔐

5. Gestió de seguretat

Control d'accés (autenticació d'usuaris, autoritzacions). Aïllament de processos (un procés no pot llegir la memòria d'un altre). Auditoria (logs). En Linux: DAC (permisos UNIX), MAC (SELinux, AppArmor). En Windows: ACL, UAC, Windows Defender.

🖥️

6. Interfície amb l'usuari

El SO proporciona una interfície per interaccionar-hi: CLI (bash, sh, zsh, PowerShell, cmd) o GUI (GNOME, KDE, Cinnamon, Windows Desktop). En entorns de servidor s'usa quasi exclusivament CLI per eficiència i menor consum de recursos.

Tema 5 — Aplicacions

Tipus d'aplicacions

🛠️

Aplicacions de sistema

Utilitats que gestionen el sistema: gestors de discos (fdisk, Disk Management), compiladors (gcc, clang), depuradors (gdb), eines de monitoratge (Nagios, Zabbix), shells (bash, PowerShell).

💼

Aplicacions d'usuari

Productivitat (Microsoft Office, LibreOffice), navegadors (Firefox, Chrome), correus (Thunderbird, Outlook), editors de codi (VS Code, Vim), bases de dades (MySQL Workbench).

🌐

Serveis / Daemons

Programes sense interfície gràfica que corren en segon pla. Apache/Nginx (web), OpenSSH (accés remot), cron (tasques programades), systemd (gestor de serveis en Linux). En Windows: Services.

🐳

Aplicacions virtualitzades / contenidors

Programes empaquetats amb les seves dependències. Docker containers, Snap (Ubuntu), Flatpak (Linux), AppImage. Avantatge: portabilitat total entre distribucions.

🔌

Controladors (Drivers)

Software de sistema que permet al SO comunicar-se amb el hardware. Corren a nivell de kernel. Exemples: driver de targeta gràfica (NVIDIA, AMD), driver de xarxa, driver d'impressora.

🧰

Middleware

Capa intermèdia entre el SO i les aplicacions: Java Virtual Machine (JVM), .NET Runtime, ODBC (connexions a BD), servidors d'aplicacions (JBoss, Tomcat). Afavoreixen la portabilitat del codi.

Llicències i tipus de llicències

El marc legal que regula l'ús, distribució i modificació del programari. Clau en decisió de sistemes empresarials.

Fonaments legals

Programari lliure vs programari propietari

La llicència és el contracte legal entre el creador del software i l'usuari. Defineix el que l'usuari pot i el que no pot fer. Com a tècnics ASIX hem de saber quina llicència té cada peça de software que instal·lem a l'empresa.

🆓

Les 4 llibertats del Programari Lliure (FSF/GNU)

Definides per Richard Stallman. Un programa és lliure si l'usuari té:

  • 🔢 Llibertat 0: Executar el programa per a qualsevol propòsit.
  • 🔢 Llibertat 1: Estudiar com funciona el programa (necessita accés al codi font).
  • 🔢 Llibertat 2: Redistribuir còpies del programa.
  • 🔢 Llibertat 3: Distribuir còpies de les versions modificades (millora la comunitat).
💡

Atenció: "lliure" fa referència a llibertat, no a preu (free as in freedom, not free as in beer). El programari lliure pot ser de pagament.

🔒

Programari propietari (privatiu)

L'usuari rep el software compilat (codi font no disponible) i accepta una llicència EULA (End-User License Agreement) que restringeix:

  • 🚫 Redistribució: no es pot copiar ni distribuir el software sense permís.
  • 🚫 Enginyeria inversa: prohibida en la majoria de llicències.
  • 🚫 Modificació: el codi font no és accessible.
  • 💰 Model de negoci: llicències per usuari, per dispositiu, per subscripció (SaaS).
Exemples propietaris

Windows (Microsoft) · macOS (Apple) · Office 365 · Adobe Creative Cloud · VMware vSphere · Oracle Database

Implicació pràctica: el "copyleft viral" de la GPL

Si una empresa incorpora codi amb llicència GPL al seu producte comercial, tot el producte resultant ha de publicar-se amb GPL. Empreses com Apple, Microsoft i moltes altres eviten deliberadament usar codi GPL en els seus productes principals, preferint llicències MIT o Apache que permeten la incorporació en software propietari sense aquesta restricció. Alternativa empresarial: LGPL (per a biblioteques) o llicències duals (OpenSSL: OpenSSL License + MIT).

7

Gestors d'arrencada

La seqüència d'arrencada completa: de l'encesa del botó al prompt de login. BIOS, UEFI, MBR, GPT, GRUB2 i Windows Boot Manager.

La seqüència d'arrencada

Del botó d'encesa al prompt de login

Entendre la seqüència completa d'arrencada és fonamental per diagnosticar errors, configurar sistemes dual-boot i gestionar entorns d'instal·lació. Aquí la seqüència detallada:

  1. Power-On: La font d'alimentació estabilitza la tensió (Power Good signal). La CPU rep un reset hardware i carrega la primera instrucció des d'una adreça fixa (0xFFFFFFF0 en x86).

  2. Firmware (BIOS/UEFI): El firmware s'executa des de la memòria flash de la placa base. Realitza el POST (Power-On Self Test): comprova CPU, RAM, dispositius. Si hi ha error crític: bips de codi o missatge en pantalla.

  3. Detecció de dispositius: El firmware enumera tots els dispositius connectats (PCI, USB, SATA, NVMe) i els inicialitza bàsicament. Construeix la taula ACPI per comunicar la topologia al SO.

  4. Selecció del dispositiu d'arrencada: Basant-se en l'ordre de boot configurat al BIOS/UEFI, selecciona el dispositiu (disc, USB, xarxa PXE…).

  5. MBR o EFI System Partition: En BIOS+MBR: llegeix els primers 512 bytes del disc (MBR), que contenen el bootloader de primera etapa. En UEFI+GPT: accedeix directament a la partició EFI (ESP, format FAT32) i executa el fitxer .efi del gestor d'arrencada.

  6. Bootloader (GRUB2 / Windows Boot Manager): Mostra el menú d'arrencada, carrega el kernel del SO a la RAM i li passa el control.

  7. Kernel: S'inicia el kernel del SO. Detecta el hardware (dmesg), munta el sistema de fitxers arrel, inicialitza els mòduls/drivers.

  8. Init / systemd (Linux) o smss.exe (Windows): El primer procés d'espai d'usuari (PID 1 en Linux). Arrenca tots els serveis del sistema en l'ordre correcte. En Linux modern: systemd. En Windows: Session Manager Subsystem.

  9. Login Manager: Es presenta la pantalla d'inici de sessió (GDM, SDDM, LightDM en Linux; Winlogon en Windows) o el prompt de terminal en servidors.

⚠️ BIOS (Legacy) — Característiques

  • 📅 Origen: IBM PC, 1981. Interfície de 16 bits en mode real.
  • 💾 MBR (Master Boot Record): 512 bytes al sector 0 del disc. Limitat a 4 particions primàries i discs < 2 TB.
  • 🔢 Direccionament: CHS (Cylinder-Head-Sector), obsolet. Limitat a 2,2 TB.
  • 🔓 Seguretat: cap verificació de la cadena d'arrencada. Vulnerable a bootkits.

✅ UEFI — L'estàndard modern

  • 📅 Origen: Intel EFI → UEFI Forum. Estàndard des de ~2006, obligatori des de Win 8.
  • 💾 GPT (GUID Partition Table): fins a 128 particions primàries, discs > 2 TB (límit teòric: 9,4 ZB).
  • 🔐 Secure Boot: verifica la signatura criptogràfica del bootloader. Prevé bootkits i rootkits.
  • 🖥️ Interfície: GUI gràfica, suport de ratolí, xarxa des del firmware, shell UEFI (EFI Shell).

🐧 GRUB2 (Grand Unified Bootloader v2)

El gestor d'arrencada estàndard de la majoria de distribucions Linux (Ubuntu, Debian, Fedora, etc.).

Tema 9 — Instal·lació

Procés d'instal·lació de SO

🪟 Instal·lació Windows 11

  1. Descarregar la ISO oficial de microsoft.com/software-download i crear un USB bootable amb Rufus o l'Media Creation Tool.

  2. A la UEFI: verificar que Secure Boot i TPM 2.0 estan actius (requisits obligatoris de Win 11).

  3. Arrencar des del USB. Seleccionar idioma, teclat, edició (Home/Pro/Server). Introduir la clau de producte (o saltar per activar posteriorment).

  4. Triar "Instal·lació personalitzada" per gestionar les particions manualment. Crear particions: EFI (100 MB), MSR (16 MB), Windows (mínim 64 GB), Recuperació (500 MB).

  5. L'instal·lador copia els fitxers (uns 20-30 minuts). El sistema reinicia diverses vegades.

  6. Configuració inicial (OOBE): compte Microsoft o local, privacitat, nom del PC, PIN.

  7. Postinstal·lació: Windows Update, drivers (Device Manager), activació.

🐧 Instal·lació Ubuntu Server 24.04 LTS

  1. Descarregar la ISO de ubuntu.com/download/server. Crear USB amb balenaEtcher o dd if=ubuntu.iso of=/dev/sdX bs=4M status=progress.

  2. Arrencar des del USB. Seleccionar idioma, teclat, tipus d'instal·lació (minimitzada o completa).

  3. Configuració de xarxa: IP estàtica recomanada per a servidors (o DHCP per a proves). Configurar proxy si cal.

  4. Particionament: guiat en disc sencer (ràpid) o manual (recomanat en producció). Esquema recomanat: /boot/efi (1 GB, FAT32), / (20+ GB, ext4), swap (= RAM si poca RAM), /home o /var separats si és servidor.

  5. Crear l'usuari administrador (no root directament). Activar OpenSSH Server per accés remot.

  6. Seleccionar snap de servidor (opcional): LAMP, Docker, Kubernetes, etc.

  7. L'instal·lador finalitza, reinicia. Postinstal·lació: apt update && apt upgrade, configuració de xarxa, SSH keys.

Tema 12 — Fitxers d'inici Linux

systemd: el gestor de serveis modern

En les distribucions Linux modernes (Ubuntu 15.04+, Debian 8+, RHEL 7+, Fedora 15+), systemd és el primer procés que s'executa (PID 1) i gestiona tots els serveis del sistema.

Conceptes clau de systemd

  • ⚙️ Unit: la unitat bàsica de systemd. Cada servei, muntatge, socket, etc. és una "unit". Definides en fitxers .service, .mount, .socket, .timer…
  • 🎯 Target: equivalent als antics runlevels. multi-user.target = CLI, graphical.target = GUI, rescue.target = mode segur.
  • 📂 Ubicació dels units: /lib/systemd/system/ (del SO), /etc/systemd/system/ (personals/override).
  • 🔗 Dependències: Requires=, Wants=, After=, Before= permeten ordenar l'arrencada dels serveis.

Fitxers de configuració importants

/etc/fstab

# <dispositiu> <punt_muntatge> <tipus_fs> <opcions> <dump> <pass>
UUID=abc123 / ext4 errors=remount-ro 0 1
UUID=def456 /boot/efi vfat umask=0077 0 1
UUID=ghi789 none swap sw 0 0
//srv/compartit /mnt/nfs nfs defaults 0 0

/etc/rc.local (deprecated, usat per compatibilitat)

Fitxer llegit en l'arrencada per executar comandos manuals. Millor alternativa modern: crear un .service de systemd amb ExecStart= i WantedBy=multi-user.target.

systemctl — Control de serveisUbuntu
# Veure l'estat general del sistema
root@host:~# systemctl status

# Llistar tots els serveis actius
root@host:~# systemctl list-units --type=service --state=active

# Gestió d'un servei concret (nginx d'exemple)
root@host:~# systemctl start nginx # Iniciar
root@host:~# systemctl stop nginx # Aturar
root@host:~# systemctl restart nginx # Reiniciar
root@host:~# systemctl reload nginx # Recarregar config (sense aturar el servei)
root@host:~# systemctl enable nginx # Activar a l'arrencada (symlink a /etc/systemd/system/)
root@host:~# systemctl disable nginx # Desactivar de l'arrencada
root@host:~# systemctl is-enabled nginx # Comprovar si s'inicia automàticament

# Veure logs del servei (journald)
root@host:~# journalctl -u nginx -f # -f = follow (en temps real)
root@host:~# journalctl -b # Logs des de l'últim arrencada
root@host:~# journalctl --since "1 hour ago"
Tema 13 — Registre de Windows

El Registre del sistema (Windows Registry)

El Registre de Windows és la base de dades jeràrquica centralitzada on el SO i les aplicacions emmagatzemen la seva configuració. Entendre'l és imprescindible per a diagnosi, auditoria i scripting avançat.

Claus principals (Hives)

ClauAbreviacióContingut
HKEY_LOCAL_MACHINEHKLMConfiguració del SO i del hardware. Afecta tots els usuaris. Els admins l'escriuen.
HKEY_CURRENT_USERHKCUConfiguració de l'usuari actiu. Preferències, aplicacions, escriptori.
HKEY_CLASSES_ROOTHKCRAssociació d'extensions de fitxer amb aplicacions. Fusió de HKLM\Software\Classes i HKCU\Software\Classes.
HKEY_USERSHKUPerfils de tots els usuaris del sistema.
HKEY_CURRENT_CONFIGHKCCConfiguració de hardware actual (en temps real). No es guarda permanentment.

Claus importants a conèixer

  • HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion
    Versió del SO, clau de producte, data d'instal·lació
  • HKLM\SYSTEM\CurrentControlSet\Services
    Configuració de tots els serveis i drivers del sistema
  • HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run
    Programes que s'executen a l'arrencada per a tots els usuaris
  • HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run
    Programes d'inici de l'usuari actual (malware sovint aquí)

Pràctica 1.7 — systemd i Registre de Windows

Crear un servei systemd personalitzat i explorar el Registre de Windows per PowerShell

🔀 Windows + Ubuntu
🐧 Ubuntu — Crear un servei systemd personalitzat
Terminal — rootUbuntu
# Crear un script que s'executarà com a servei
root@host:~# cat > /usr/local/bin/hola-servei.sh << 'EOF'
#!/bin/bash
while true; do
echo "$(date): Servei en execució" >> /var/log/hola-servei.log
sleep 60
done
EOF
root@host:~# chmod +x /usr/local/bin/hola-servei.sh

# Crear el fitxer .service
root@host:~# cat > /etc/systemd/system/hola-servei.service << 'EOF'
[Unit]
Description=El meu primer servei systemd
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/hola-servei.sh
Restart=on-failure
User=nobody

[Install]
WantedBy=multi-user.target
EOF

# Recarregar systemd, habilitar i iniciar el servei
root@host:~# systemctl daemon-reload
root@host:~# systemctl enable --now hola-servei.service
root@host:~# systemctl status hola-servei.service

Actualització i manteniment de controladors

Drivers en Windows (Device Manager, DISM, PowerShell) i mòduls del kernel a Linux (lsmod, modprobe, DKMS).

Controladors de dispositiu (Drivers)

Un controlador (driver) és un component de software que permet al SO comunicar-se amb un dispositiu de hardware específic. Operen al nivell de kernel i una driver defectuós pot causar kernel panic (Linux) o pantalla blava BSoD (Windows).

🪟

Gestió de drivers a Windows

  • 🖥️ Administrador de dispositius (devmgmt.msc): eina gràfica. Dispositius amb ⚠️ indiquen driver faltant o conflicte.
  • ✅ Firma digital WHQL: Microsoft certifica els drivers perquè superin proves de compatibilitat. Un driver sense firma requereix Secure Boot desactivat.
  • ⬇️ Fonts: Windows Update (automàtic), web del fabricant (NVidia, AMD, Intel), Driver Store.
  • ↩️ Rollback: si un driver nou causa problemes, pots tornar a la versió anterior des del Device Manager → Propietats → Controlador → Retrocedeix el controlador.
PowerShellWindows
# Llistar tots els drivers instal·lats
PS C:\> Get-WindowsDriver -Online | Sort-Object ClassName | Format-Table ClassName, OriginalFileName, Version

# Buscar drivers de dispositius sense instal·lar
PS C:\> Get-WmiObject Win32_PnPEntity | Where-Object{$_.ConfigManagerErrorCode -ne 0}

# Exportar (backup) tots els drivers de tercers
PS C:\> Export-WindowsDriver -Online -Destination C:\DriversBackup

# Forçar instal·lació d'un driver .inf
PS C:\> pnputil /add-driver C:\driver.inf /install

Mòduls del kernel a Linux

A Linux, els drivers es carreguen com a mòduls del kernel (fitxers .ko — kernel object). Poden carregar-se i descarregar-se dinàmicament sense reiniciar.

Terminal — rootUbuntu
# Llistar tots els mòduls carregats
root@host:~# lsmod
Module Size Used by
nvidia 56774720 0
kvm_intel 471040 0

# Informació d'un mòdul específic
root@host:~# modinfo kvm_intel

# Carregar un mòdul manualment
root@host:~# modprobe kvm_intel

# Descarregar un mòdul
root@host:~# modprobe -r kvm_intel

# Fer que un mòdul es carregui a l'arrencada
root@host:~# echo "kvm_intel" >> /etc/modules-load.d/kvm.conf

# Bloquejar un mòdul (blacklist, per conflictes)
root@host:~# echo "blacklist nouveau" >> /etc/modprobe.d/blacklist.conf
# Drivers addicionals recomanats (Ubuntu GUI)
root@host:~# ubuntu-drivers devices
root@host:~# ubuntu-drivers autoinstall

DKMS — Dynamic Kernel Module Support

Quan el kernel de Linux s'actualitza, els mòduls compilats per a la versió antiga deixen de funcionar. DKMS (Dynamic Kernel Module Support) recompila automàticament els mòduls de tercers (com els drivers NVIDIA o VirtualBox) quan s'actualitza el kernel. Sense DKMS, caldria reinstal·lar manualment cada driver de tercers en cada actualització del kernel.
