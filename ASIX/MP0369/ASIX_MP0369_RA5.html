Arquitectura d'Active Directory

Estructura jeràrquica d'Active Directory

Active Directory Domain Services (AD DS) és el servei de directori de Microsoft que centralitza la gestió d'identitats, ordinadors, recursos i polítiques en una xarxa corporativa. Entendre la seva jerarquia és fonamental per administrar permisos a escala.

Jerarquia d'objectes AD

🌍 Forest: techcorp.com
├─ 🏢 Domini: techcorp.com (domini arrel)
│ ├─ 📦 OU: Usuaris
│ │ ├─ 👤 Usuari: joan.garcia ← GPO: Pol_Usuaris
│ │ └─ 👤 Usuari: anna.puig
│ ├─ 📦 OU: Equips
│ │ ├─ 💻 Equip: PC-IT-001 ← GPO: Pol_Equips_IT
│ │ └─ 💻 Equip: PC-RRHH-001
│ ├─ 📦 OU: Grups
│ │ ├─ 👥 Grup: G_IT_Admins (Seguretat Global)
│ │ └─ 👥 Grup: G_Lectura_Docs (Distribució)
│ └─ 📦 OU: Servidors
│ └─ 🖥️ Servidor: SRV-FILE-01 ← GPO: Pol_Servidors
└─ 🏢 Domini fill: barcelona.techcorp.com
🌍

Forest i Domini

Forest: col·lecció de dominis AD que comparteixen esquema i catàleg global. La frontera de seguretat màxima.

Domini: unitat administrativa amb la seva pròpia base de dades AD (NTDS.dit), controladors de domini (DC) i polítiques. Els comptes d'un domini no accedeixen automàticament als recursos d'un altre.

📦

OU — Unitat Organitzativa

Les OU (Organizational Units) son contenidors dins d'un domini per organitzar objectes (usuaris, equips, grups, servidors). Permeten:

  • 🎯 Aplicar GPO de forma granular a subconjunts d'objectes
  • 🔑 Delegar permisos d'administració per OU
  • 🗂️ Reflectir l'estructura organitzativa de l'empresa
Grups d'Active Directory

Tipus i àmbits de grups AD: AGDLP

AD defineix grups per tipus (Seguretat vs Distribució) i per àmbit (Local de Domini, Global, Universal). L'estratègia AGDLP és la millor pràctica per assignar accés a recursos.

ÀmbitMembres acceptatsPot ser membre deAssignació de permisosÚs típic

💡
Estratègia AGDLP (Account → Global → Domain Local → Permission):

Millor pràctica per assignar permisos en entorns de domini: Afegir el compte d'usuari a un Grup Global, afegir el grup global a un grup de Domini Local, i assignar el Permís al grup de domini local sobre el recurs. Exemple: Joan.Garcia → G_IT_Admins → DL_Escriptura_Projectes → Permís NTFS sobre \\srv\projectes

Protocol d'autenticació

Kerberos: autenticació al domini sense enviar contrasenyes

Kerberos és el protocol d'autenticació de Windows i Active Directory. En lloc d'enviar la contrasenya per la xarxa (com feia NTLM), usa un sistema de tiquets criptogràfics que permet l'autenticació sense revelar credencials.

Flux d'autenticació Kerberos (simplificat)

1. L'usuari introdueix la contrasenya al PC client
→
2. El client sol·licita un TGT (Ticket Granting Ticket) al KDC (Domain Controller)
→
3. El KDC verifica les credencials i emet el TGT (xifrat, vàlid 10h)
→
4. Per accedir a un recurs, el client presenta el TGT i rep un Service Ticket específic
→
5. El servidor verifica el Service Ticket i atorga accés. La contrasenya mai surt del client.

🪟 Windows Server — PowerShell
Crear estructura AD completa🪟 Windows Server
# Import mòdul AD
PS> Import-Module ActiveDirectory
# Crear estructura d'OUs
PS> New-ADOrganizationalUnit -Name "TechCorp" -Path "DC=techcorp,DC=com"
PS> @("Usuaris","Equips","Grups","Servidors") | ForEach-Object { New-ADOrganizationalUnit -Name $_ -Path "OU=TechCorp,DC=techcorp,DC=com" }

PS> $pass = ConvertTo-SecureString "P@ssw0rd2025!" -AsPlainText -Force
PS> New-ADUser -Name "Joan Garcia" -SamAccountName "joan.garcia" -UserPrincipalName "joan.garcia@techcorp.com" -Path "OU=Usuaris,OU=TechCorp,DC=techcorp,DC=com" -AccountPassword $pass -Enabled $true -Department "IT" -Title "Tècnic ASIX"
# Crear grups i afegir membres (estratègia AGDLP)
PS> New-ADGroup -Name "G_IT_Admins" -GroupScope Global -GroupCategory Security -Path "OU=Grups,OU=TechCorp,DC=techcorp,DC=com"
PS> New-ADGroup -Name "DL_Escriptura_Projectes" -GroupScope DomainLocal -GroupCategory Security -Path "OU=Grups,OU=TechCorp,DC=techcorp,DC=com"
PS> Add-ADGroupMember -Identity "G_IT_Admins" -Members "joan.garcia"
PS> Add-ADGroupMember -Identity "DL_Escriptura_Projectes" -Members "G_IT_Admins"
# Verificar estructura (LDAP query)
PS> Get-ADUser -Filter * -SearchBase "OU=Usuaris,OU=TechCorp,DC=techcorp,DC=com" -Properties Department,Title | Format-Table Name,SamAccountName,Department

  • Implementa AGDLP: Crea grups Globals (G_IT, G_RRHH, G_Finances) i grups Domain Local per a cada nivell d'accés (DL_Lectura, DL_Escriptura, DL_Control_Total). Vincula seguint AGDLP.
  • Ubuntu al domini: Uneix una VM Ubuntu al domini usant realm join. Comprova que pots fer su - joan.garcia@techcorp.com i iniciar sessió amb les credencials AD.
  • Verifica els tiquets Kerberos: Executa klist a Windows i a Ubuntu després del login. Comprova que hi ha un TGT vàlid i interpreta els camps (Principal, Server, End Time, Flags).

📜

Regles clau dels permisos NTFS

  • ➕ Acumulació: els permisos de múltiples grups s'ACUMULEN (OR). Si ets membre de G1 (Lectura) i G2 (Escriptura), tens Lectura + Escriptura.
  • 🚫 Denegar té prioritat: un permís "Denegar" explícit anul·la qualsevol "Permetre", independentment de quin grup l'atorga. Usa-ho amb precaució.
  • 📋 Explícit > Heretat: un permís assignat directament a un objecte (explícit) té prioritat sobre el permís heretat del pare, fins i tot si el heretat és "Denegar".
  • 👑 Propietari del fitxer: el propietari (NTFS Owner) sempre pot canviar els permisos del seu propi objecte, fins i tot si no té "Control Total".
⚠️
Trampa del "Denegar":

Afegir un usuari al grup "Tots" amb permís Denegar bloquejarà fins i tot als Administradors. En lloc de Denegar, és millor simplement NO afegir el permís de Permetre.

Herència de permisos

Herència: com es propaguen els permisos als subdirectoris

Per defecte, els permisos NTFS s'hereten cap als subdirectoris i fitxers fills. Això simplifica l'administració però pot causar problemes si no s'entén correctament. Saber quan trencar l'herència és una habilitat clau.

🔽

Indicadors d'herència (icacls)

  • (OI) Object Inherit — Els fitxers fills hereten el permís
  • (CI) Container Inherit — Els subdirectoris hereten
  • (IO) Inherit Only — Sols per a fils, no aplica al directori actual
  • (NP) No Propagate — No propaga als néts
  • (I) Inherited — Aquest permís és heretat del pare
✂️

Trencar l'herència

Quan trenquem l'herència d'un directori, els permisos del pare deixen de propagar-se. Podem triar:

  • 📋 Convertir en explícits: els permisos heretats es copien com a explícits (pots modificar-los).
  • 🗑️ Eliminar heretats: comença sense permisos i en configures de nous des de zero.
Quan trencar l'herència

Directori que necessita restriccions molt més estrictes que el seu pare. Ex: C:\Empresa\RRHH\Nomines — el pare permet lectura a tots els empleats, però Nòmines ha de ser solo per a RRHH.

🔍

Permisos efectius

El permís efectiu és el resultat final de combinar: permisos explícits + heretats, de tots els grups, tenint en compte les denegacions.

Calcular permisos efectius🪟
# PowerShell — permisos efectius
PS> Get-Acl C:\Empresa\RRHH | Format-List
# GUI: clic dret → Propietats → Seguretat → Avançat → Accés efectiu
# icacls
> icacls C:\Empresa\RRHH
TECHCORP\G_RRHH:(OI)(CI)(M)
BUILTIN\Administrators:(OI)(CI)(F)
TECHCORP\joan.garcia:(I)(R) ← Heretat

Configura una estructura de permisos complexa per a l'empresa TechCorp i verifica els permisos efectius

🪟 Windows
Configuració de permisos NTFS complexos🪟 Windows
# Crear estructura
PS> New-Item -ItemType Directory -Path C:\Empresa\{RRHH,IT,Finances,Comú}

Samba — Interoperabilitat Windows/Linux via SMB

Servidor Samba, compartir carpetes, autenticació, integració amb AD i gestió de permisos SMB+NTFS.

Protocol SMB i Samba

Samba: el pont entre Linux i Windows

Samba és la implementació de codi obert del protocol SMB/CIFS (Server Message Block / Common Internet File System) per a Linux. Permet que un servidor Linux comparteixi fitxers i impressores amb clients Windows (i viceversa), i fins i tot actuar com a Controlador de Domini Active Directory (Samba 4).

🦷

Modes de Samba

  • 📂 Servidor de fitxers (Standalone): comparteix carpetes sense domini. Autenticació pròpia (smbpasswd).
  • 👥 Membre de domini (Domain Member): s'uneix a un domini AD Windows. Usa les credencials d'AD per autenticar. El més comú en entorns mixtos.
  • 🏢 Controlador de domini (Samba 4 AD DC): Samba actua com a DC d'Active Directory pur. Compatible amb clients Windows.
📦

Instal·lació i estructura

Instal·lació Samba🐧 Ubuntu
$ sudo apt install samba samba-common -y
# Fitxers importants:
/etc/samba/smb.conf ← Configuració principal
/var/lib/samba/ ← Bases de dades
/var/log/samba/ ← Logs
$ sudo systemctl enable --now smbd nmbd
$ sudo smbstatus # Connexions actives
🪟

Versions del protocol SMB

  • SMBv1 Obsolet i INSEGUR. Vulnerabilitat EternalBlue (WannaCry). Deshabilitar SEMPRE.
  • SMBv2 Windows Vista+. Millores de rendiment. Acceptable però preferir v3.
  • SMBv3 Windows 8/Server 2012+. Xifrat de dades en trànsit. Recomanat.
🔒 Deshabilitar SMBv1

Set-SmbServerConfiguration -EnableSMB1Protocol $false

Configuració de smb.conf

El fitxer smb.conf: seccions i paràmetres clau

/etc/samba/smb.conf — Configuració completa comentada🦷 Samba
######################## SECCIÓ [global] ########################
[global]
workgroup = TECHCORP # Nom del grup de treball o domini NT
server string = Servidor Samba TechCorp
netbios name = SRV-SAMBA # Nom NetBIOS del servidor
security = ads # ads=domain AD, user=standalone, share=públic
realm = TECHCORP.COM # Domini Kerberos (en mode ads)
log file = /var/log/samba/%m.log
max log size = 1000
min protocol = SMB2 # ⚠️ DESHABILITAR SMBv1!
server signing = mandatory # Signatura digital obligatòria (anti-MitM)
smb encrypt = required # Xifrat SMBv3 obligatori
#################### CARPETA PÚBLICA (tothom lliu i) ####################
[public]
path = /srv/samba/public
browseable = yes
read only = yes
guest ok = yes # Accés sense autenticació
#################### CARPETA DEPARTAMENT IT ####################
[it_dept]
path = /srv/samba/it
browseable = yes
read only = no
valid users = @G_IT_Admins, @G_IT_Support # @ = grup Samba/AD
write list = @G_IT_Admins # Sols admins poden escriure
create mask = 0660 # Permisos dels nous fitxers
directory mask = 0770 # Permisos dels nous directoris
force group = +it_admins # Tots els fitxers creats pertanyen a aquest grup
#################### HOME DIRECTORIES ####################
[homes]
browseable = no # No apareix a la llista de compartits
read only = no
valid users = %S # %S = nom del share (= nom d'usuari)
# Verificar configuració
$ testparm # Valida el smb.conf sense errors
$ testparm -s | grep -A5 "\[it_dept\]"

Configura Samba en mode standalone amb compartits per a 3 departaments i users específics

🦷 Samba🔀 Clients
  • Prepara l'estructura: A l'Ubuntu server, crea els directoris /srv/samba/{public,it,rrhh,finances}. Crea els grups Linux it_grp, rrhh_grp, finances_grp. Crea 6 usuaris (2 per departament). Configura permisos Linux (POSIX) correctes.

NFS — Network File System

Compartir sistemes de fitxers entre màquines Linux/Unix. NFS4, exports, permisos i seguretat.

NFS vs Samba

NFS: el protocol de fitxers de xarxa per a Unix/Linux

NFS (Network File System) és el protocol estàndard per compartir sistemes de fitxers en entorns Linux/Unix. A diferència de Samba (dissenyat per a interoperabilitat amb Windows), NFS és natiu de Linux i ofereix millor integració amb els permisos POSIX i millor rendiment en entorns homogenis.

⚠️
Problema de UID/GID amb NFS:

NFS basa els permisos en l'UID i GID del client. Si l'usuari joan té UID 1001 al servidor però UID 1003 al client, el servidor li negarà l'accés o li donarà accés incorrecte. Solucions: (a) sincronitzar UID/GID (LDAP/AD), (b) usar NFSv4 + Kerberos amb mapeig d'identitats, (c) NIS (obsolet).

🔒
NFSv4: millores de seguretat clau
  • Suport natiu per a Kerberos (sec=krb5)
  • Pseudo-sistema de fitxers unificat
  • Suport per ACLs POSIX i NFSv4
  • Millor gestió de la delegació de fitxers
  • Un sol port (2049/TCP) vs NFSv3 multipuerto

Permisos de xarxa, locals i permisos efectius

Com interactuen els permisos de compartit (Share Permissions) amb els permisos NTFS locals. Càlcul del permís efectiu final.

Permisos de compartit vs NTFS

El model de dos nivells: Share Permissions + NTFS

Quan un usuari accedeix a un recurs compartit per xarxa a Windows, el sistema aplica dos filtres de permisos en sèrie: primer els Share Permissions (permisos del recurs compartit, a nivell de xarxa) i després els permisos NTFS (a nivell de fitxer). El permís final efectiu és la intersecció més restrictiva dels dos.

🌐

Share Permissions (Permisos de xarxa)

S'apliquen sols quan s'accedeix per xarxa. Si accedeixes localment al servidor, no s'apliquen.

  • 👁️ Lectura: llistar i llegir fitxers
  • ✏️ Canvi: llegir + escriure + eliminar
  • 👑 Control Total: tot + canviar permisos de share
Millor pràctica

Dona "Control Total" a "Tots" als Share Permissions. Controla l'accés detallat exclusivament amb NTFS. Menys confusió, un sol punt de control.

🔢

Càlcul del permís efectiu final (xarxa)

Permís efectiu final = min(Share Permissions, NTFS Permissions)

Cas 2LecturaControl TotalLectura ← Share és el limitant
Cas 3CanviModificarModificar (equivalent)
Cas 4Control TotalControl TotalControl Total
Cas 5 ⚠️Control Total "Tots"Denegar a "joan"Denegar — el Denegar NTFS guanya sempre
💡

Accés local vs accés per xarxa: quan l'Administrador accedeix directament al servidor (sessió local o RDP), els Share Permissions NO s'apliquen. Sols els NTFS. Per això un admin local té accés complert tot i que el Share Permission li doni sols "Lectura".

Delegació de permisos

Delegació de control en Active Directory: administrar per OU sense ser Domain Admin. Principi de mínim privilegi.

Delegació en AD

Delegar tasques d'administració sense donar privilegis de Domain Admin

La delegació de control permet assignar a usuaris o grups la capacitat d'administrar objectes específics d'Active Directory (una OU, un tipus d'objecte, un atribut concret) sense necessitar el grup privilegiat Domain Admins. És la implementació del principi de mínim privilegi a escala de directori.

🎯

Casos d'ús de la delegació

  • 👤 HelpDesk: pot restablir contrasenyes a la OU d'Usuaris, però no crear ni eliminar comptes.
  • 🏢 Administrador de dept.: gestiona els equips i usuaris de la seva OU (ex: OU=Barcelona), sense tocar la OU de Madrid.
  • 📧 Operador de grups: pot afegir/eliminar membres de grups de distribució, sense poder crear ni eliminar grups.
  • 📋 Actualitzar atributs: l'equip de RRHH pot actualitzar el número de telèfon i departament dels usuaris, sense accés a contrasenyes ni permisos.
Delegació de control — PowerShell AD🪟 Windows Server
# Delegar "Restablir contrasenyes" al grup G_HelpDesk
# sobre la OU Usuaris
PS> $ouDN = "OU=Usuaris,OU=TechCorp,DC=techcorp,DC=com"
PS> $group = Get-ADGroup "G_HelpDesk"
# GUID del dret "Reset Password" a AD
PS> $guidResetPwd = [GUID]"00299570-246d-11d0-a768-00aa006e0529"
PS> $ace = New-Object System.DirectoryServices.ActiveDirectoryAccessRule($group.SID, "ExtendedRight", "Allow", $guidResetPwd, "Descendents", [GUID]"bf967aba-0de6-11d0-a285-00aa003049e2")
PS> $acl = Get-Acl "AD:\$ouDN"
PS> $acl.AddAccessRule($ace)
PS> Set-Acl "AD:\$ouDN" $acl
# Alternativa: Wizard gràfic
# ADUC → clic dret sobre OU → "Delegar control..."
⚠️
La delegació no és visible fàcilment

Les delegacions en AD s'emmagatzemen a les ACLs dels objectes AD (no als grups). Usa l'eina dsacls o ADSI Edit per auditar qui té quins permisos delegats.

Directives de Grup (GPO)

Group Policy Objects: creació, vinculació, herència, àmbit, filtratge, drets d'usuari, directives de seguretat i plantilles administratives.

Fonaments de GPO

Què és una GPO i com s'aplica?

Un Group Policy Object (GPO) és un conjunt de configuracions del sistema operatiu (usuari i/o equip) que s'aplica automàticament als objectes d'AD (usuaris, equips) dins l'àmbit vinculat. Permet administrar centenars d'equips de forma centralitzada.

Ordre d'aplicació: LSDOU

LSDOU: l'ordre d'aplicació de les GPO

Les GPO s'apliquen en un ordre determinat. L'ordre importa perquè les GPO aplicades LAST WIN (les últimes sobreescriuen les anteriors, excepte en cas de conflicte o Enforce).

🔒

Enforce (No Override)

Marca una GPO com a "Enforce". Impedeix que les GPO d'OUs fills la sobreescriguin. Útil per a polítiques de seguretat corporatives que cap OU filla pot relaxar (ex: política de contrasenyes, bloqueig de panell de control).

Exemple

GPO "Pol_Seguretat_Global" vinculada al Domini amb Enforce: bloqueja el CMD i el Panell de Control per a tots els usuaris, independentment de les GPO de les OUs filles.

🛑

Block Inheritance

Una OU pot "bloquejar" l'herència de GPO dels seus pares. Les GPO del Domini i les OUs pare no s'apliquen a aquesta OU. Excepció: les GPO amb Enforce s'apliquen igualment.

Cas d'ús

OU=Kioscos: bloqueja l'herència per tenir configuració completament diferent a la resta del domini.

🔍

Security Filtering

Per defecte, una GPO s'aplica a "Authenticated Users" (tots). Pots filtrar per un grup específic: la GPO sols s'aplica als membres d'aquell grup. Permet una granularitat molt fina.

Exemple avançat

GPO "Restriccions_Usuaris_Normals": vinculada al Domini, però filtre de seguretat = G_Usuaris_Estàndard. No s'aplica a G_IT_Admins.

Creació i gestió de GPOs

Crear, configurar i vincular GPOs: casos d'ús reals

GPO — PowerShell i GPMC🪟 Windows Server
# Instal·lar mòdul de GPO
PS> Install-WindowsFeature GPMC
PS> Import-Module GroupPolicy
# ─── Crear GPOs ───
PS> New-GPO -Name "Pol_Seguretat_Global" -Domain "techcorp.com" -Comment "Política de seguretat corporativa"
PS> New-GPO -Name "Pol_Escriptori_Usuaris" -Domain "techcorp.com"
# ─── Vincular GPO a una OU ───
PS> New-GPLink -Name "Pol_Seguretat_Global" -Target "DC=techcorp,DC=com" -Enforced Yes
PS> New-GPLink -Name "Pol_Escriptori_Usuaris" -Target "OU=Usuaris,OU=TechCorp,DC=techcorp,DC=com"
# ─── Configurar ajustos via GPO Registry ───
# Desactivar CMD per a usuaris
PS> Set-GPRegistryValue -Name "Pol_Seguretat_Global" -Key "HKCU\Software\Policies\Microsoft\Windows\System" -ValueName "DisableCMD" -Type DWord -Value 1
# Configurar fons d'escriptori corporatiu
PS> Set-GPRegistryValue -Name "Pol_Escriptori_Usuaris" -Key "HKCU\Software\Microsoft\Windows\CurrentVersion\Policies\System" -ValueName "Wallpaper" -Type String -Value "\\srv-file\public\wallpaper.jpg"
# ─── Security Filtering ───
# Eliminar "Authenticated Users" (per defecte) i afegir sols G_Usuaris_Estàndard
PS> Set-GPPermissions -Name "Pol_Escriptori_Usuaris" -PermissionLevel None -TargetName "Authenticated Users" -TargetType Group
PS> Set-GPPermissions -Name "Pol_Escriptori_Usuaris" -PermissionLevel GpoApply -TargetName "G_Usuaris_Estàndard" -TargetType Group
# ─── Diagnosi ───
> gpupdate /force # Forçar actualització de GPO al client
> gpresult /R # RSoP: quines GPO s'apliquen a l'usuari actual
> gpresult /H gporeport.html # Informe HTML detallat (Resultant Set of Policy)
PS> Get-GPResultantSetOfPolicy -ReportType Html -Path C:\gpo_report.html
Drets d'usuari i Directives de Seguretat

User Rights Assignment i Security Settings

Dins d'una GPO, la secció Configuració de Windows → Configuració de Seguretat → Directrius Locals conté dues seccions crítiques: els Drets d'Usuari (User Rights Assignment) i les Opcions de Seguretat.

🎫

Drets d'usuari (User Rights Assignment)

Controlen el que un compte pot fer al sistema, independentment dels permisos d'objecte:

  • 🖥️ Iniciar sessió localment: qui pot fer login físic a l'equip.
  • 🔌 Accés per xarxa a l'equip: qui pot connectar-se per xarxa.
  • 🔧 Gestionar el registre d'auditoria: qui pot netejar els logs d'esdeveniments.
  • 🚀 Canviar la zona horària: útil per a quioscos (impedeix canviar-la).
  • 💾 Fer còpies de seguretat: qui pot usar l'API de backup sense permisos de fitxer.
⚙️

Opcions de seguretat importants

  • 🔒 Compte: Reanomenar Administrador: canvia el nom del compte Administrator (menys evident per a atacants).
  • 🖥️ Inici interactiu: No mostrar últim nom d'usuari: no mostra el darrer usuari al login.
  • 🔐 Xarxa: Nivell d'autenticació LAN Manager: forçar NTLMv2 o Kerberos, rebutjar LM i NTLM.
  • 📵 Dispositius: Impedir que els usuaris instal·lin impressores: seguretat als equips compartits.
Plantilles administratives

Plantilles administratives (ADMX/ADML): configuració del registre via GPO

Les plantilles administratives (fitxers ADMX) permeten gestionar configuracions de Windows i aplicacions (Office, Chrome, Firefox...) via GPO sense editar el registre manualment. S'apliquen escrivint valors al registre de Windows als clients.

📄

Plantilles ADMX incorporades

  • 🔒 Windows Update: forçar actualitzacions automàtiques des del servidor WSUS intern.
  • 🛡️ Windows Defender: configurar la protecció anti-malware centralment.
  • 🔐 BitLocker: forçar el xifrat de disc i gestionar les claus de recuperació a AD.
  • 📡 Remote Desktop: habilitar/deshabilitar RDP, forçar autenticació NLA.
🌐

Plantilles de tercers (Chrome, Firefox, Office)

Moltes aplicacions proporcionen plantilles ADMX pròpies:

  • 🌐 Google Chrome ADMX: forçar pàgina d'inici, extensions permeses, deshabilitar incògnit.
  • 📊 Office 365 ADMX: configurar macro security, aplicar plantilles, deshabilitar telemetria.
  • 🦊 Firefox ADMX: configurar proxy, certificats, pàgina d'inici corporativa.
💻

Scripts via GPO

  • 🚀 Inici de sessió (Logon): muntar unitats de xarxa, sincronitzar fitxers, actualitzar configuració.
  • 🔌 Tancament de sessió (Logoff): desar perfil, desconnectar unitats, auditoria.
  • 🖥️ Inici/Apagada de l'equip: instal·lar/desinstal·lar programari, actualitzar inventari.

🔀 Ambdós + Samba + NFS

TechCorp S.L. t'ha encarregat la implementació completa de la infraestructura d'accés. Disposes d'un Windows Server 2022 (DC), un Ubuntu Server (servidor de fitxers Samba/NFS) i clients Windows i Ubuntu.

📋 FASE 1 — Estructura Active Directory
  • Estructura d'OUs: TechCorp → {Usuaris/{IT, RRHH, Finances}, Equips/{Portàtils, Escriptoris}, Servidors, Grups/{Seguretat, Distribució}, Servei_Comptes}.
  • Comptes: 2 usuaris per departament + 1 compte de servei per Samba (svc_samba) + grup G_IT, G_RRHH, G_Finances + grups DL per a cada nivell d'accés.
  • AGDLP complet: implementa l'estratègia per als recursos de xarxa. Documenta el diagrama de relació Usuaris → GG → DL → Recurs.
📋 FASE 2 — Servidor de fitxers Samba integrat amb AD
  • Uneix l'Ubuntu Server al domini com a membre (security = ads). Configura SSSD per a autenticació AD.
  • Shares Samba: [it_dept] (G_IT → RW, autenticació AD), [rrhh] (G_RRHH → RW, ocult), [finances] (G_Finances → RW), [public] (tothom → RO, guest ok), [homes] (cada usuari el seu).
  • Verifica l'accés: Des de clients Windows, munta les unitats amb net use. Des de clients Ubuntu, munta amb mount -t cifs. Comprova que els permisos AD es respecten.
📋 FASE 3 — Servidor NFS per als servidors Linux
  • Exports NFS: /srv/nfs/homes (tota la subxarxa 192.168.1.0/24, rw, root_squash), /srv/nfs/backups (sols IP del servidor de backup, rw, no_root_squash), /srv/nfs/iso (qualsevol, ro, all_squash).
  • Automatitza el muntatge als clients Ubuntu amb fstab (NFSv4, _netdev). Comprova que es munta a l'arrencada.
📋 FASE 4 — Permisos, herència i delegació
  • NTFS + Share: Per al share it_dept: Share = Tots → CT. NTFS = G_IT_Admins → Control Total (heredat), G_IT_Support → Lectura i Execució. Subdirectori /it_dept/Confidencial: trenca herència, sols G_IT_Admins.
  • Delegació: G_HelpDesk → Restablir contrasenyes a OU=Usuaris. G_IT_Support → Unir equips al domini. G_RRHH_Leads → Gestionar usuaris a OU=RRHH (crear, modificar, no eliminar).
  • Verifica els permisos efectius de 3 usuaris de perfils diferent i documenta la matriu completa de "qui pot fer què a cada recurs".
📋 FASE 5 — GPO corporativa completa
  • 3 GPOs: Pol_Seguretat_Global (Domini, Enforced), Pol_Usuaris_Estàndard (OU Usuaris, filtrat per G_Usuaris), Pol_Servidors (OU Servidors, forçar auditoria i deshabilitar serveis innecessaris).
  • Script Logon: Munta Z: = home personal, W: = Públic. Crea un log de sessions a \\srv-file\it_dept\login_audit.log amb data, hora i nom d'usuari.
  • RSoP complet: Genera un informe HTML per a cada tipus d'usuari (usuari normal, admin IT, usuari RRHH). Analitza les diferències.
📋 FASE 6 — Documentació tècnica
  • Diagrama d'arquitectura: Esquema visual de tota la infraestructura: AD (DC), servidor Samba, servidor NFS, clients, xarxa. Inclou protocols (SMB, NFS, Kerberos, LDAP).
  • Matriu de permisos: Taula completa (usuaris × recursos) amb els permisos efectius de cada combinació.
  • Informe de seguretat: Llista de mesures de seguretat implementades (SMBv1 deshabilitat, NTLMv2 forçat, root_squash NFS, Enforce GPO, delegació mínima) amb la justificació de cada una.

Resum del RA5

El que heu après

Repàs dels conceptes clau del RA5 i la seva aplicació professional.

🌐

Active Directory

Jerarquia: Forest → Domini → OU → Objectes. Grups: Global, Domain Local, Universal. AGDLP: A→GG→DL→Permís. Kerberos: TGT + Service Ticket, sense contrasenya per xarxa. realm join per a Linux.

🦷

Samba & NFS

Samba: SMBv3, smb.conf, valid users, write list, force group. Deshabilitar SMBv1. NFS: /etc/exports, rw/ro, root_squash, NFSv4+Kerberos. Clau: NFS per Linux↔Linux, Samba per interoperabilitat Windows.

🔑

Permisos

NTFS: permisos s'acumulen, Denegar guanya, explícit > heretat. Share + NTFS: permís final = min(Share, NTFS). Millor pràctica: Tots→CT als Share, controla tot per NTFS. Efectius: eina "Accés efectiu".

📋

GPO

LSDOU: Local→Site→Domain→OU (last wins). Enforce: cap OU filla pot sobreescriure. Block Inh.: bloqueja GPO pare. Security Filter: aplicar sols a un grup. RSoP: gpresult /H. Delegació: mínim privilegi per OU.
