Administració d'usuaris i grups locals

Conceptes, tipus de comptes, identificadors únics (SID/UID/GID) i gestió des de GUI i línia de comandes.

Fonaments

Què és un compte d'usuari?

Un compte d'usuari és un objecte del sistema operatiu que representa una identitat digital. Conté credencials (nom + contrasenya), permisos, configuració personal i historial de sessions. Tota acció al sistema queda associada a un compte — és la base de la traçabilitat i la seguretat.

🪪

Identificadors únics

Cada compte té un identificador numèric únic que el SO usa internament. El nom d'usuari és una etiqueta llegible per humans, però el SO treballa amb l'ID.

  • 🪟 SID (Windows): Security Identifier. Format: S-1-5-21-XXXXXXXX-XXXXXXXX-XXXXXXXX-RID. Únic i permanent — no canvia si es reanomena el compte.
  • 🐧 UID (Linux): User Identifier, enter de 32 bits. root=0, sistemes=1-999, usuaris=1000+.
  • 🐧 GID (Linux): Group Identifier. Cada usuari té un grup primari (igual que el seu UID per defecte) i pot pertànyer a grups secundaris.
🪟

Tipus de comptes a Windows

  • 👑 Administrador: accés complet al sistema. Pot instal·lar programes, modificar configuració global i gestionar altres usuaris.
  • 👤 Usuari estàndard: accés limitat. Pot usar aplicacions i modificar la seva configuració personal, però no la del sistema.
  • 👻 Convidat (Guest): accés molt restringit. Deshabilitat per defecte a Windows 10/11. Sense perfil persistent.
🐧

Tipus de comptes a Linux

  • 👑 root (UID 0): superusuari. Accés absolut. Usar root directament és perillós — Ubuntu el desactiva i usa sudo.
  • 🔧 Usuaris del sistema (1–999): comptes per a serveis (www-data, syslog, nobody). No fan login interactiu.
  • 👤 Usuaris normals (1000+): comptes de persones reals. Primer usuari creat: UID 1000. Poden usar sudo si estan al grup sudo.
Bases de dades d'usuaris

On s'emmagatzemen els comptes?

Windows i Linux utilitzen sistemes completament diferents per emmagatzemar la informació dels comptes. Conèixer-los és fonamental per a la diagnosi i la seguretat.

🗄️

Windows: SAM i Active Directory

En sistemes locals (sense domini), els comptes s'emmagatzemen a la base de dades SAM:

  • 📂 Ubicació: C:\Windows\System32\config\SAM
  • 🔐 Format: les contrasenyes es guarden com a hash NTLM (MD4 del password Unicode). No es guarda la contrasenya en text pla.
  • 🔒 Bloqueig: el fitxer SAM és bloquejat pel kernel mentre Windows és actiu. No es pot llegir directament.
  • 🌐 En domini: els comptes es centralitzen a Active Directory (AD DS) al controlador de domini.
📄

Linux: /etc/passwd i /etc/shadow

Linux usa fitxers de text plans (llegibles amb cat):

  • 📄 /etc/passwd: llista de tots els comptes. Llegible per tothom. Format: nom:x:UID:GID:GECOS:home:shell
  • 🔐 /etc/shadow: contrasenyes en hash (SHA-512). Només llegible per root. Format: nom:hash:última_canvi:min:max:avís:inactiu:expiració
  • 👥 /etc/group: grups i els seus membres. Format: nom_grup:x:GID:membres
  • 🔑 /etc/gshadow: contrasenyes de grups (rarament usat). Protegit com /etc/shadow.
Estructura dels fitxers d'usuaris a Linux 🐧 Ubuntu
asix@ubuntu:~$ cat /etc/passwd | head -5
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
bin:x:2:2:bin:/bin:/usr/sbin/nologin
...
# Format: nom:contrasenya(x=shadow):UID:GID:GECOS:home:shell
asix@ubuntu:~$ sudo cat /etc/shadow | grep asix
asix:$6$salt$hash_sha512...:19800:0:99999:7:::
# Camps del shadow: últim canvi(dies des 1/1/1970):min:max:avís:inactiu:expiració
asix@ubuntu:~$ id asix
uid=1000(asix) gid=1000(asix) groups=1000(asix),4(adm),27(sudo),1001(docker)

Usuaris predeterminats

Comptes predeterminats del sistema

Tots els sistemes operatius creen una sèrie de comptes predeterminats durant la instal·lació. Conèixer-los, les seves funcions i els seus riscos de seguretat és fonamental per a un administrador.

Compte Sistema SID/UID Funció Estat per defecte Risc si s'activa
Administrator Windows ...-500 Compte d'administrador integrat. Accés total. Deshabilitada 🔴 Molt alt — objectiu d'atacs de força bruta per nom conegut
Guest Windows ...-501 Accés temporal sense perfil persistent. Deshabilitada 🟡 Mig — accés anònim a recursos compartits
DefaultAccount Windows ...-503 Compte intern del sistema (Microsoft Store apps). Deshabilitada 🟢 Baix
WDAGUtilityAccount Windows ...-504 Windows Defender App Guard (sandboxing). Deshabilitada 🟢 Baix

⚠️
Bones pràctiques de seguretat sobre comptes predeterminats:
  • Windows: Reanomena el compte Administrator (no es pot eliminar, però es pot reanomenar). Mantén-lo desactivat i usa un compte administrador amb nom no evident.
  • Linux: Mantén el login directe de root desactivat via SSH (PermitRootLogin no a /etc/ssh/sshd_config). Usa sudo amb usuaris del grup sudo.
  • Revisa periòdicament els comptes actius. Desactiva qualsevol compte que no es faci servir.

Conceptes de grups

Per a què serveixen els grups?

Assignar permisos usuari per usuari és inviable en entorns professionals. Els grups permeten gestionar permisos col·lectivament: s'assignen permisos al grup, i tots els seus membres els hereten automàticament. Canviar els permisos d'un grup afecta immediatament tots els seus membres.

Lab 2.1 — Creació i gestió de grups. Assignació de membres

Estructura de grups per a un departament d'IT d'empresa

🔀 Ambdós

Creareu una estructura de grups que reflecteixi l'organigrama d'un departament IT: IT_Admins (administradors), IT_Helpdesk (suport), IT_ReadOnly (auditoria, accés de lectura).

🪟 Windows — PowerShell
PowerShell — Gestió de grups 🪟 Windows
# Crear grups locals
PS C:\> New-LocalGroup -Name "IT_Admins" -Description "Administradors del departament IT"
PS C:\> New-LocalGroup -Name "IT_Helpdesk" -Description "Equip de suport tècnic"
PS C:\> New-LocalGroup -Name "IT_ReadOnly" -Description "Accés de lectura per auditoria"
# Afegir membres als grups
PS C:\> Add-LocalGroupMember -Group "IT_Admins" -Member "tec_asix"
PS C:\> Add-LocalGroupMember -Group "IT_Helpdesk" -Member "jdoe","alumne1"
# Llistar membres d'un grup
PS C:\> Get-LocalGroupMember -Group "IT_Helpdesk"
ObjectClass Name PrincipalSource
----------- ---- ---------------
User DESKTOP-XXX\alumne1 Local
User DESKTOP-XXX\jdoe Local
# Llistar tots els grups locals
PS C:\> Get-LocalGroup | Sort-Object Name
🐧 Ubuntu — Terminal
Terminal Ubuntu — Gestió de grups 🐧 Ubuntu
# Crear grups
asix@ubuntu:~$ sudo groupadd it_admins
asix@ubuntu:~$ sudo groupadd it_helpdesk
asix@ubuntu:~$ sudo groupadd it_readonly
# Afegir usuaris a grups (sense treure'ls dels actuals)
asix@ubuntu:~$ sudo usermod -aG it_admins tec_asix
# -a = append (afegir sense eliminar grups existents), -G = grups addicionals
asix@ubuntu:~$ sudo usermod -aG it_helpdesk jdoe,alumne1
# Veure membres d'un grup
asix@ubuntu:~$ getent group it_helpdesk

# Tots els grups del sistema
asix@ubuntu:~$ cat /etc/group | grep "it_"
# IMPORTANT: els canvis de grup no apliquen fins a nou login del usuari
asix@ubuntu:~$ newgrp it_admins # aplicar sense logout

  • Elimina un usuari d'un grup sense eliminar-lo del sistema: Remove-LocalGroupMember / gpasswd -d usuari grup.

Seguretat de comptes d'usuari

Polítiques de contrasenyes, bloqueig de comptes, caducitat, auditoria i principi de mínim privilegi.

Polítiques de contrasenyes

Política de contrasenyes: teoria i implementació

Una contrasenya feble és la porta d'entrada més comuna als sistemes. La política de contrasenyes defineix els requisits mínims que ha de complir tota contrasenya del sistema. Un administrador ha de conèixer com configurar-les, no només aplicar-les.

🔑

Paràmetres de política de contrasenyes

  • 📏 Longitud mínima: recomanació actual NIST: ≥12 caràcters. No usar màxim arbitrari.
  • 🔤 Complexitat: majúscules, minúscules, dígits, símbols. Windows: opció "La contrasenya ha de complir requisits de complexitat".
  • 📅 Edat màxima: dies que pot estar activa una contrasenya. NIST ja no recomana canvis periòdics forçats (creen hàbits pitjors).
  • 🔄 Historial: nombre de contrasenyes anteriors que no es poden reutilitzar. Evita el cicle ABC→ABC→ABC.
  • ⏳ Edat mínima: temps mínim que s'ha de mantenir una contrasenya. Evita canviar 24 vegades seguidament per tornar a l'original.
🚫

Bloqueig de comptes

Protegeix contra atacs de força bruta limitant els intents d'autenticació fallits:

  • 🔢 Límit d'intents: nombre d'errors consecutius permesos. Recomanació: 5-10 intents.
  • ⏱️ Durada del bloqueig: temps que el compte roman bloquejat. 0 = manual per admin. 30 min = automàtic.
  • 🔁 Finestra d'observació: temps durant el qual es compten els intents fallits.
💡

DoS intern: un bloqueig agressiu (3 intents) pot ser usat per atacants per bloquejar usuaris legítims intencionadament.

Implementació a Windows

Política de seguretat local a Windows

Windows gestiona la política de seguretat local a través de secpol.msc (GUI) o net accounts / PowerShell (línia de comandes). En entorns de domini, les GPO (Group Policy Objects) substitueixen la configuració local.

PowerShell / cmd — Política de comptes Windows 🪟 Windows
# Veure configuració actual de política de comptes
C:\> net accounts
Force user logoff how long after time expires?: Never
Minimum password age (days): 0
Maximum password age (days): 42
Minimum password length: 0
Length of password history maintained: None
Lockout threshold: Never
Lockout duration (minutes): 30
# Configurar política robusta
C:\> net accounts /minpwlen:12 /maxpwage:90 /minpwage:1 /uniquepw:5
The command completed successfully.
C:\> net accounts /lockoutthreshold:5 /lockoutduration:30 /lockoutwindow:30
The command completed successfully.
# Bloquejar / desbloquejar compte manualment
PS C:\> Disable-LocalUser -Name "jdoe"
PS C:\> Enable-LocalUser -Name "jdoe"
# Forçar canvi de contrasenya en proper login
PS C:\> Set-LocalUser -Name "jdoe" -PasswordNeverExpires $false
PS C:\> net user jdoe /logonpasswordchg:yes
Implementació a Linux

PAM i chage — política de contrasenyes a Ubuntu

Linux usa el mòdul PAM (Pluggable Authentication Modules) per gestionar l'autenticació. La política de contrasenyes es configura al fitxer /etc/pam.d/common-password i la caducitat amb l'eina chage.

Ubuntu — PAM i chage 🐧 Ubuntu
# Instal·lar el mòdul de qualitat de contrasenyes
asix@ubuntu:~$ sudo apt install libpam-pwquality -y
# Editar /etc/security/pwquality.conf per configurar la política
asix@ubuntu:~$ sudo nano /etc/security/pwquality.conf
# Contingut recomanat del fitxer:
minlen = 12 # Longitud mínima
dcredit = -1 # Almenys 1 dígit (-N = mínim N)
ucredit = -1 # Almenys 1 majúscula
lcredit = -1 # Almenys 1 minúscula
ocredit = -1 # Almenys 1 caràcter especial
maxrepeat = 3 # Màxim 3 caràcters iguals seguits
usercheck = 1 # No pot contenir el nom d'usuari
# Caducitat de contrasenyes amb chage
asix@ubuntu:~$ sudo chage -l jdoe
Last password change : Jun 01, 2025
Password expires : never
Password inactive : never
Account expires : never
Minimum number of days between password change : 0
Maximum number of days between password change : 99999
# Configurar caducitat: màx 90 dies, avís 7 dies abans, mínim 1 dia
asix@ubuntu:~$ sudo chage -M 90 -m 1 -W 7 -I 30 jdoe
# -M=màx dies, -m=min dies, -W=dies d'avís, -I=inactivitat fins bloqueig
# Forçar canvi en proper login
asix@ubuntu:~$ sudo chage -d 0 jdoe
# Bloquejar / desbloquejar compte
asix@ubuntu:~$ sudo usermod -L jdoe # Lock
asix@ubuntu:~$ sudo usermod -U jdoe # Unlock
# Verificar estat del compte (! = bloquejat)
asix@ubuntu:~$ sudo passwd -S jdoe
jdoe L 2025-06-01 1 90 7 30

Lab 3.1 — Implementar política de seguretat corporativa

Configura una política de seguretat completa per a un entorn empresarial

🔀 Ambdós

L'empresa "TechCorp S.L." et demana implementar una política de seguretat de comptes. Els requisits són: contrasenyes de mínim 12 caràcters amb complexitat, historial de 5 contrasenyes, caducitat a 90 dies, bloqueig als 5 intents fallits i desbloqueig automàtic als 30 minuts.

  • Reflexiona: Quina diferència hi ha entre un compte bloquejat (lockout per intents fallits) i un compte desactivat (disabled per l'administrador)? Pot un compte bloquejat desbloquejar-se sol?

Perfils d'usuari a Windows

Estructura i tipus de perfils a Windows

Un perfil d'usuari és el conjunt de configuració personal (escriptori, documents, configuració d'aplicacions, variables d'entorn) associat a un compte. A Windows, cada usuari té el seu propi perfil aïllat, de manera que els canvis d'un usuari no afecten els altres.

📁

Estructura del perfil de Windows

El perfil s'emmagatzema a C:\Users\<nom_usuari>\ i conté:

  • 🖥️ Desktop\ — Arxius i dreceres de l'escriptori
  • 📄 Documents\ — Carpeta personal de documents
  • ⬇️ Downloads\ — Descàrregues
  • 🖼️ Pictures\, Videos\, Music\ — Multimèdia
  • ⚙️ AppData\ — Configuració d'aplicacions (hidden)
  •   ├ AppData\Local\ — Dades locals (no s'exporten al perfil itinerant)
  •   ├ AppData\Roaming\ — Dades que sí s'exporten al servidor
  •   └ AppData\LocalLow\ — Dades d'aplicacions de confiança baixa
🏠

Perfil local

Emmagatzemat a la màquina local. Es crea automàticament en el primer login. Si canvies d'equip, el perfil no es transfereix. Ideal per a estacions fixes amb usuari dedicat.

🌐

Perfil itinerant (Roaming)

S'emmagatzema en un servidor de xarxa. Quan l'usuari fa login a qualsevol equip del domini, rep el seu perfil actualitzat. Requereix domini Active Directory. Sincronitza AppData\Roaming\.

🔒

Perfil obligatori

Versió de només lectura d'un perfil itinerant. L'usuari pot fer canvis durant la sessió, però tot es descarta en tancar. S'usa en quioscos, biblioteques i aules informàtiques.

Configuració del protocol TCP/IP

Adreçament IPv4/IPv6, màscara de subxarxa, gateway, DHCP vs estàtica, i eines de diagnòstic.

Fonaments TCP/IP

El model TCP/IP i l'adreçament

La configuració de xarxa és un dels rols principals d'un administrador de sistemes. TCP/IP és el protocol universal que permet la comunicació entre equips. Configurar-lo incorrectament fa que l'equip no pugui comunicar-se amb cap recurs de xarxa.

🏠

Adreça IP

Identificador únic de cada interfície de xarxa dins d'una xarxa.

  • 🔢 IPv4: 32 bits en notació decimal puntada. Ex: 192.168.1.100. ≈4.300 milions d'adreces.
  • 🌐 IPv6: 128 bits en hexadecimal. Ex: 2001:db8::1. 3,4×10³⁸ adreces. Obligatori en infraestructures modernes.
  • 🔁 Loopback: 127.0.0.1 (IPv4) / ::1 (IPv6). Apunta a l'equip mateix. Mai surt per la interfície física.
🎭

Màscara de subxarxa

Defineix quina part de l'adreça IP identifica la xarxa i quina identifica l'host:

Exemple de càlcul

IP: 192.168.1.100
Màscara: 255.255.255.0 = /24
Xarxa: 192.168.1.0
Rang hosts: 192.168.1.1 – .254
Broadcast: 192.168.1.255
Hosts disponibles: 254

🚪

Gateway (porta d'enllaç)

Adreça IP del router que connecta la xarxa local amb xarxes externes (Internet). Tots els paquets amb destí fora de la subxarxa local s'envien al gateway.

Configuració típica

IP equip: 192.168.1.100/24
Gateway: 192.168.1.1 (el router)
DNS primari: 8.8.8.8 (Google) o el del ISP
DNS secundari: 8.8.4.4

Rangs d'adreces IPv4

💡
APIPA (169.254.x.x) — diagnosi important:

Si un equip té una adreça 169.254.x.x, significa que no ha pogut obtenir IP del servidor DHCP. Causes: el servidor DHCP no és accessible, cable de xarxa desconnectat, o servei DHCP aturat. Aquesta adreça NO permet comunicació amb Internet ni amb altres xarxes.

DHCP vs Estàtica

Assignació dinàmica (DHCP) vs estàtica

🔄

DHCP — Dynamic Host Configuration Protocol

El servidor DHCP assigna automàticament IP, màscara, gateway i DNS a cada equip que es connecta.

  • ✅ Avantatges: zero configuració a l'equip client, gestió centralitzada, evita conflictes d'IP.
  • ❌ Inconvenients: l'IP pot canviar (lease renewal). Problemàtic per a servidors o impresores.
  • ⏱️ Lease time: temps que el client manté l'IP. En renovar, pot ser la mateixa o diferent.
Procés DORA

Discover → Offer → Request → Acknowledge. El client fa broadcast, el servidor ofereix una IP, el client la sol·licita formalment i el servidor confirma.

📌

IP estàtica

L'administrador configura manualment l'adreça IP directament a l'equip. Sempre la mateixa.

  • ✅ Avantatges: IP sempre la mateixa, previsible. Imprescindible per a servidors, impressores, NAS, càmeres IP.
  • ❌ Inconvenients: gestió manual equip per equip, risc de conflictes d'IP si es duplica.
  • 💡 Alternativa: DHCP reservation — l'IP és "estàtica" però la gestiona el DHCP per la MAC address del dispositiu.

Configuració de la resolució de noms

DNS, fitxers hosts, NSS (Name Service Switch), resolució local i proves de diagnosi.

El sistema DNS

Com funciona la resolució de noms?

Quan escrivim www.google.com, el sistema ha de traduir aquest nom a una adreça IP per poder enviar paquets. Aquest procés s'anomena resolució de noms i segueix un ordre de consulta definit per la configuració del sistema.

1. Aplicació sol·licita resolució de "www.google.com"
El navegador, ping, curl... fan una crida al sistema
2. Consulta la caché DNS local del sistema
Windows: ipconfig /displaydns · Linux: systemd-resolve --statistics
3. Consulta el fitxer /etc/hosts (o C:\Windows\System32\drivers\etc\hosts)
Resolució estàtica local — sempre té prioritat sobre DNS
4. Consulta els servidors DNS configurats (8.8.8.8, etc.)
Resolució recursiva: DNS local → DNS arrel → TLD → autoritatiu
5. Resposta: 142.250.185.14 → l'aplicació es connecta a aquesta IP
El resultat es guarda a la caché local per futures consultes
📋

El fitxer /etc/hosts

Fitxer de resolució local estàtica. Té prioritat sobre DNS (per defecte). Usat per a: redirigir dominis, bloquejar publicitat, definir hostnames locals.

/etc/hosts — Linux o C:\Windows\System32\drivers\etc\hosts 🔀 Ambdós
# Format: IP hostname aliases
127.0.0.1 localhost
127.0.1.1 asix-ubuntu
192.168.1.10 servidor1.local servidor1
192.168.1.20 impresora.local printer
# Bloquejar un domini (redirigir al loopback)
0.0.0.0 ads.example.com

NSS i /etc/nsswitch.conf (Linux)

El fitxer /etc/nsswitch.conf defineix l'ordre de consulta per a cada tipus de resolució:

/etc/nsswitch.conf 🐧 Ubuntu

# files = /etc/hosts, dns = servidor DNS, mdns = mDNS (Bonjour)
# Resolució de comptes d'usuari
passwd: files systemd

# Si es treballa en domini LDAP:
passwd: files ldap systemd

Arquitectura de xarxa a Linux modern (Ubuntu 18+)

Capa d'abstracció declarativa. L'administrador escriu YAML; Netplan genera la config per al backend.

Implementa TCP/IP, gestiona les interfícies físiques i virtuals.
Hardware: NIC Ethernet, Wi-Fi, loopback
Les interfícies físiques. Noms moderns: ens33, enp2s0, wlan0, lo...

Optimització per a portàtils i arxius sense connexió

Gestió d'energia a Windows

Plans d'energia i configuració per a portàtils

Els ordinadors portàtils requereixen una gestió d'energia específica per equilibrar rendiment i autonomia. Windows ofereix plans d'energia configurables i modes d'estalvi que afecten el comportament de la CPU, la pantalla i la connectivitat de xarxa.

⚡

Plans d'energia de Windows

  • ⚡ Màxim rendiment: CPU al 100%, sense limitació. Per a treballs intensius. Bateria: molt baixa.
  • ⚖️ Equilibrat (recomanat): ajusta automàticament CPU i pantalla. Bon equilibri rendiment/bateria.
  • 🌱 Estalvi d'energia: CPU limitada, pantalla obscura. Màxima autonomia per a tasques lleugeres.
😴

Estats de suspensió

  • 💤 Suspensió (Sleep/S3): RAM amb corrent (estat conservat). Reanudació instant. Bateria: consumeix poc.
  • ❄️ Hibernació (Hibernate/S4): contingut de RAM al disc (hiberfil.sys). Equip completament apagat. Reanudació en 15-30s.
  • 🔋 Suspensió híbrida: combina S3 i S4. Si la bateria s'esgota durant la suspensió, el sistema es recupera de disc.
🐧

Gestió d'energia a Ubuntu

  • ⚡ TLP: eina per a gestió d'energia avançada en portàtils Linux. sudo apt install tlp.
  • 📊 powertop: analitza consums i suggereix optimitzacions. sudo powertop.
  • 🔋 upower: daemon de gestió de bateria. upower -i /org/freedesktop/UPower/devices/battery_BAT0.
Arxius sense connexió (Offline Files)

Carpetes de xarxa disponibles sense connexió

La funcionalitat Offline Files (arxius fora de línia) permet que els usuaris d'un portàtil continuïn treballant amb arxius de carpetes compartides de xarxa quan no hi ha connexió disponible. Els canvis es sincronitzen automàticament quan es recupera la connexió.

🔄

Com funciona Offline Files

  1. L'administrador comparteix una carpeta al servidor i habilita la caché del costat del client.
  2. L'usuari marca la carpeta de xarxa per a ús fora de línia (Clic dret → Sempre disponible fora de línia).
  3. Windows sincronitza una còpia local a %SystemRoot%\CSC (Client-Side Cache).
  4. Quan l'usuari treballa sense xarxa, treballa sobre la còpia local sense notar cap diferència.
  5. En reconnectar-se, Windows sincronitza automàticament els canvis (ambdues direccions).
⚠️

Conflictes de sincronització

Si un fitxer és modificat tant localment (usuari en itinerància) com al servidor (un company), es produeix un conflicte. Windows avisa i demana a l'usuari quina versió conservar: la local, la del servidor, o ambdues amb noms diferents.

🔁

Alternativa Linux: rsync i sshfs

A Linux, la sincronització offline es pot implementar amb rsync (còpia diferencial) o unison (sincronització bidireccional). sshfs munta carpetes remotes via SSH. rclone sincronitza amb núvol (Google Drive, S3, etc.).

El que heu après

Repàs dels conceptes clau del RA2 i la seva aplicació professional com a tècnics ASIX.

👥

Identitats

Windows: SID, SAM, Get-LocalUser, New-LocalUser, comptes predeterminats. Linux: UID/GID, /etc/passwd, /etc/shadow, useradd, usermod. Clau: el SID no canvia en reanomenar un compte.

🔒

Seguretat

Windows: net accounts, secpol.msc, Disable-LocalUser. Linux: PAM + pwquality, chage, faillock, usermod -L/-U. Clau: bloqueig ≠ desactivació. NIST recomana 12+ car. i sense canvi periòdic forçat.

🌐

Xarxa TCP/IP

Windows: New-NetIPAddress, Set-DnsClientServerAddress, ipconfig/all, tracert. Linux: ip addr, nmcli, Netplan YAML, dig, resolvectl. Clau: 169.254.x.x = DHCP fallat. Diagnosi per capes.

💻

Portàtils

Windows: powercfg, Offline Files (CSC), plans d'energia. Linux: TLP, powertop, rsync, sshfs. Clau: Offline Files sincronitza automàticament. rsync és l'equivalent Linux per a sincronització diferencial.
