Estructura client/servidor

El model que fonaments tota la infraestructura IT empresarial moderna.

Fonaments

Del model Peer-to-Peer al model Client/Servidor

Entendre per quin motiu les empreses adopten el model client/servidor és el punt de partida per comprendre la necessitat del Domini. Un grup de treball (workgroup) pot funcionar per a 5 usuaris; per a 50 és un caos.

🔀

Peer-to-Peer (Grup de Treball)

Cada equip és alhora client i servidor. No hi ha control central.

  • 👤 Comptes locals: cada equip té la seva pròpia base de dades d'usuaris. Si l'usuari té 10 equips, té 10 comptes.
  • 🔑 Contrasenyes descentralitzades: cada màquina la gestiona de manera independent. El canvi de contrasenya no es propaga.
  • 📁 Recursos compartits manualment: cal configurar cada compartició manualment a cada equip.
  • ⚠️ Límit pràctic: ~10-20 equips. Més enllà és ingobernable en termes de seguretat i manteniment.
🏢

Client/Servidor (Domini)

Un servidor central (Domain Controller) gestiona tots els recursos i identitats de la xarxa.

  • ✅ Autenticació centralitzada: un sol compte permet accedir a tots els recursos del domini (SSO — Single Sign-On).
  • 🔐 Polítiques centralitzades (GPO): les polítiques de seguretat, programari i configuració s'apliquen des del servidor a tots els clients.
  • 📊 Directori central (AD): catàleg de tots els recursos: usuaris, equips, impressores, aplicacions.
  • 🔄 Escalabilitat: des de 10 fins a milions d'objectes. Microsoft AD suporta fins a 2 bilions d'objectes per domini.
🏆
Avantatges del model client/servidor en l'empresa:
  • Seguretat uniforme: una política de contrasenyes s'aplica a tots alhora. Un compte compromès es bloqueja centralment.
  • Gestió simplificada: 1 admin pot gestionar 500 usuaris des d'una consola. Sense domini, caldrien 500 intervencions locals.
  • Recuperació davant desastres: els perfils i dades estan al servidor → si el PC de l'usuari s'espatlla, res es perd.
  • Auditoria i compliance: tots els accessos queden registrats al servidor. Imprescindible per a ISO 27001, ENS, RGPD.
  • Desplegament de software: el software s'instal·la des del servidor a tots els clients (GPO + MSI, SCCM, Intune).

2

Protocol LDAP

Lightweight Directory Access Protocol: el protocol estàndard per accedir i gestionar serveis de directori. La base d'Active Directory, OpenLDAP, Google Workspace i molts altres.

Protocol i model de dades

Què és LDAP i com estructura la informació

LDAP (RFC 4511) és un protocol lleuger basat en TCP/IP per accedir a serveis de directori. Un directori és una base de dades jeràrquica optimitzada per a lectures freqüents i escriptures poc freqüents (a l'inrevés d'una BD SQL). Active Directory DS usa LDAP com a protocol d'accés principal.

🌐

Ports LDAP

  • 📡 Port 389 (LDAP): port estàndard, comunicació en clar (no xifrada).
  • 🔐 Port 636 (LDAPS): LDAP sobre SSL/TLS. Comunicació xifrada. Obligatori en producció.
  • 🔒 Port 3268 (GC): Global Catalog en AD. Port 3269 per a GCSS (xifrat).
🌳

Jerarquia del directori

  • 🌲 DIT (Directory Information Tree): l'arbre complet del directori.
  • 📁 Entry (objecte): cada node de l'arbre. Un usuari, grup, equip, OU…
  • 🔖 DN (Distinguished Name): la "ruta completa" que identifica de manera única cada objecte.
  • 🏷️ Attribute: cada camp de dades d'un objecte (cn, mail, userPrincipalName, memberOf…).
🔤

Atributs clau

  • dc — Domain Component (part del domini)
  • ou — Organizational Unit (contenidor)
  • cn — Common Name (nom de l'objecte)
  • sn — Surname (cognom)
  • mail — Adreça de correu
  • userPrincipalName — UPN (login@domini)

Exemple d'arbre LDAP per al domini asix.local

# Arrel del domini (Base DN)
dc=asix,dc=local
│
├── ou=Usuaris,dc=asix,dc=local
│ ├── cn=Joan Garcia,ou=Usuaris,dc=asix,dc=local
│ ├── cn=Maria López,ou=Usuaris,dc=asix,dc=local
│ └── ou=Informàtica,ou=Usuaris,dc=asix,dc=local
│ └── cn=Pere Martínez,ou=Informàtica,ou=Usuaris,dc=asix,dc=local
│
├── ou=Grups,dc=asix,dc=local
│ ├── cn=Admins TI,ou=Grups,dc=asix,dc=local

│

🔑
Entendre els DN (Distinguished Name) és crític:

El DN és la "adreça única" de cada objecte al directori. Es llegeix de dreta a esquerra: primer el domini (arrel), després les OUs fins a arribar a l'objecte. Exemple: cn=Joan Garcia,ou=Informàtica,ou=Usuaris,dc=asix,dc=local significa: l'objecte "Joan Garcia" que es troba a la OU "Informàtica", dins de la OU "Usuaris", al domini "asix.local". Qualsevol crida LDAP necessita el DN complet o un filtre per localitzar l'objecte.

Exemple d'entrada LDIF — Compte d'usuari LDAP

objectClass: top
objectClass: person
objectClass: organizationalPerson

cn: Joan Garcia
sn: Garcia
givenName: Joan
uid: jgarcia
mail: jgarcia@asix.local
telephoneNumber: +34 93 123 45 67
userPassword: {SSHA}9/8i7oBhKkOjZXUgD... # hash SSHA de la contrasenya
memberOf: cn=Admins TI,ou=Grups,dc=asix,dc=local

Pràctica 4.2 — Consultes LDAP amb ldapsearch i Get-ADObject

Consultar el directori amb LDAP des de línia de comandos (Ubuntu) i PowerShell (Windows)

🔀 Windows + Ubuntu

Aprendrem a fer cerques al directori LDAP per identificar objectes i els seus atributs, tant des de Windows com des de Linux.

🐧 Ubuntu — ldapsearch (consultar AD des de Linux)
Terminal — ldap-utilsUbuntu
# Instal·lar les eines LDAP
root@host:~# apt install ldap-utils

# Cerca bàsica: tots els usuaris del domini
user@host:~$ ldapsearch -H ldap://10.0.0.1 \
-D "administrator@asix.local" -W \
-b "ou=Usuaris,dc=asix,dc=local" \
"(objectClass=user)" cn mail sAMAccountName

# Cercar un usuari específic per nom
user@host:~$ ldapsearch -H ldap://10.0.0.1 -D "administrator@asix.local" -W \
-b "dc=asix,dc=local" "(cn=Joan Garcia)"

# Comptar tots els usuaris del domini
user@host:~$ ldapsearch -H ldap://10.0.0.1 -D "administrator@asix.local" -W \
-b "dc=asix,dc=local" "(objectClass=user)" -z 0 | grep "^dn:" | wc -l
🪟 PowerShell — Módulo ActiveDirectory
PowerShell — ActiveDirectory ModulePowerShell
# Importar el mòdul (instal·lat amb RSAT)
PS C:\> Import-Module ActiveDirectory

# Consultar un usuari amb tots els seus atributs LDAP
PS C:\> Get-ADUser -Identity "jgarcia" -Properties *

# Llistar tots els usuaris amb nom, UPN i OU
PS C:\> Get-ADUser -Filter * -Properties DisplayName, EmailAddress |
Select-Object Name, SamAccountName, UserPrincipalName, DistinguishedName

# Cercar usuaris d'una OU específica (filtre LDAP directe)
PS C:\> Get-ADUser -Filter * -SearchBase "ou=Informàtica,ou=Usuaris,dc=asix,dc=local"

# Cercar usuaris amb compte desactivat
PS C:\> Search-ADAccount -AccountDisabled -UsersOnly

# Veure les OUs del domini en format arbre
PS C:\> Get-ADOrganizationalUnit -Filter * | Format-Table Name, DistinguishedName

(2) Esquema manual de l'arbre LDAP del domini creat (com el de la teoria però amb els objectes reals).

Concepte de domini, subdominis i Active Directory

Arquitectura d'un bosc AD: boscos, dominis, arbre de dominis, relacions de confiança i unitats organitzatives.

Arquitectura AD

L'arquitectura d'Active Directory Domain Services

Active Directory no és simplement un repositori d'usuaris. És una arquitectura de tres nivells jeràrquics que permet escalar des d'una PYME fins a una multinacional amb milers de dominis distribuïts per tot el món.

🌲

Bosc (Forest)

El contenidor de màxim nivell. Un bosc agrupa un o més arbres de dominis que comparteixen el Global Catalog, l'schema AD i una relació de confiança transitiva. Tots els dominis d'un bosc confien entre ells.

Exemple empresarial

Una empresa cotitzada (Grup ACME Corp.) té un bosc: acmecorp.com. Tots els seus dominis fills hi confien.

🌳

Arbre de dominis (Tree)

Un arbre és un conjunt de dominis amb un espai de noms DNS continu i jeràrquic. Els dominis fills hereten part de les polítiques del pare i tenen relació de confiança bidireccional transitiva.

Exemple jerarquia

acmecorp.com (domini arrel)
espanya.acmecorp.com (fill)
barcelona.espanya.acmecorp.com (nét)

🏢

Domini (Domain)

La unitat administrativa bàsica d'AD. Té el seu propi DC, la seva base de dades d'objectes i les seves GPOs. Límit de seguretat entre departaments/empreses d'un bosc.

Requisits mínims d'un domini

1 DC amb Windows Server + Rol AD DS
Servei DNS configurat per al domini
IP estàtica al DC
Nom FQDN del domini (p.ex. asix.local)

Unitats Organitzatives

Unitats Organitzatives (OUs): l'eina d'organització

Les OUs (Organizational Units) són els contenidors dins d'un domini. A diferència dels grups, les OUs serveixen per delegar l'administració i aplicar GPOs, no per a permisos de recursos.

📐 Estratègies de disseny d'OUs

EstratègiaOrganitzar per…Quan usar-la
Per ubicació geogràficaSeu/Oficina físicaEmpreses multinacionals amb polítiques per regió
Per departamentVendes, RRHH, TI, Finances…Empresa mitja. GPOs per departament (p.ex. bloqueig USB a RRHH)
Per funció/rolServidors, Portàtils, Impressores…GPOs per tipus de dispositiu (WindowsUpdate, seguretat)
Híbrida (recomanada)Primer ubicació, dins departamentLa més usada en entorns reals. Màxima flexibilitat per a GPOs.
⚠️

OUs vs Grups: mai confondre'ls. Les OUs serveixen per organitzar objectes i aplicar GPOs. Els Grups serveixen per assignar permisos a recursos. Un usuari pot ser en una OU i en múltiples grups alhora.

🔑 Rols FSMO — Els 5 operadors de màster d'operacions

En un domini AD hi ha rols especials que s'han de tenir en compte en entorns multi-DC:

Pràctica 4.3 — Crear l'estructura d'OUs del domini

Dissenyar i crear l'arbre d'OUs per a l'empresa "TechCorp SL" (fictícia)

🪟 Windows Server

Crearem una estructura d'OUs que representi l'empresa TechCorp SL, amb seus a Barcelona i Madrid, i departaments de Vendes, IT i RRHH.

🪟 PowerShell — Creació d'OUs en massa
PowerShell — AD ModulePowerShell
# Crear OUs principals (sedes)
PS C:\> New-ADOrganizationalUnit -Name "Barcelona" -Path "dc=asix,dc=local" -ProtectedFromAccidentalDeletion $true
PS C:\> New-ADOrganizationalUnit -Name "Madrid" -Path "dc=asix,dc=local" -ProtectedFromAccidentalDeletion $true

# Crear sub-OUs per departament dins de cada seu
PS C:\> @("Usuaris","Equips","Grups") | ForEach-Object {
New-ADOrganizationalUnit -Name $_ -Path "ou=Barcelona,dc=asix,dc=local"
New-ADOrganizationalUnit -Name $_ -Path "ou=Madrid,dc=asix,dc=local"
}

# Crear sub-OUs de departaments dins Usuaris
PS C:\> @("Vendes","IT","RRHH","Finances") | ForEach-Object {
New-ADOrganizationalUnit -Name $_ -Path "ou=Usuaris,ou=Barcelona,dc=asix,dc=local"
}

# Verificar l'estructura creada
PS C:\> Get-ADOrganizationalUnit -Filter * | Sort-Object DistinguishedName | Format-Table Name, DistinguishedName -AutoSize
📋 Lliurament: Captura de la sortida del darrer Get-ADOrganizationalUnit mostrant totes les OUs. A més, obre la consola gràfica Active Directory Users and Computers (ADUC) i captura la vista en arbre de les OUs creades. Justifica per quin motiu s'usa -ProtectedFromAccidentalDeletion $true.

Tema 5 — Contrasenyes i bloquejos

Polítiques de contrasenyes i bloquejos de compte (PSO)

AD DS permet definir polítiques de contrasenyes a dos nivells: la Default Domain Policy (s'aplica a tot el domini) i les Fine-Grained Password Policies (PSO) (polítiques de grano fi, per a grups específics — disponibles des de Windows Server 2008).

🔑

Paràmetres de la política de contrasenyes

ParàmetreRecomanació NIST / Entorn empresa
Longitud mínima≥ 14 caràcters (NIST 800-63B)
ComplexitatMajúscules + minúscules + números + símbols
HistorialRecordar les últimes 24 contrasenyes
Edat màxima90 dies (NIST ara recomana sense expiració si es detecta compromís)
Edat mínima1 dia (evita canviar 24 vegades per tornar a la original)
Xifratge reversible❌ Desactivat sempre (emmagatzema en clar = vulnerabilitat crítica)
🔒

Paràmetres de bloqueig de compte

ParàmetreValor recomanat
Llindar de bloqueig5 intents fallits
Finestra d'observació30 minuts
Durada del bloqueig30 min (o 0 = fins que l'admin desbloqueja)
🚨

Atac de bloqueig massiu (Account Lockout DoS): un atacant pot intentar bloquejar sistemàticament tots els comptes d'usuari del domini, causant una denegació de servei. La compte Administrator (RID-500) és immune a les polítiques de bloqueig per aquest motiu.

🎓
Fine-Grained Password Policies (PSO) — Polítiques de grano fi

Permet assignar una política de contrasenyes diferent a grups d'usuaris específics, sense afectar la política per defecte del domini. Exemples d'ús: el grup "Domain Admins" té una política amb contrasenya de ≥20 caràcters i sense expiració + MFA obligatori; els usuaris normals tenen la política estàndard del domini. Les PSOs s'apliquen als objectes de tipus msDS-PasswordSettings i s'assignen a grups de seguretat globals.

Tema 6 — Comptes d'usuari i equips

Comptes d'usuari i comptes d'equip al domini

👤

Comptes d'usuari del domini

A diferència dels comptes locals (emmagatzemats a la SAM de cada equip), els comptes de domini s'emmagatzemen a la base de dades AD (NTDS.dit) del DC i s'autentiquen via Kerberos.

  • 🔤 SamAccountName: l'antic nom NetBIOS (≤ 20 caràcters). P.ex.: jgarcia
  • 📧 UserPrincipalName (UPN): format email, usat per login modern. P.ex.: jgarcia@asix.local
  • 🆔 SID (Security Identifier): identificador únic intern. Immutable. Exemple: S-1-5-21-1234567890-9876543210-1122334455-1103
  • 🔗 Compte vs Perfil vs Dades: un compte pot ser eliminat i recreat amb el mateix nom, però el nou compte tindrà un SID diferent i perdrà tots els permisos del compte antic.
💻

Comptes d'equip del domini

Cada PC que s'uneix al domini crea automàticament un compte d'equip (Computer Account) a AD. Aquest compte permet que l'equip s'autentiqui al domini.

  • 🔑 Contrasenya del compte d'equip: es renova automàticament cada 30 dies (Netlogon Secure Channel). L'equip la gestiona sol.
  • 📋 Naming convention: el compte d'equip es diu igual que el hostname del PC, amb un $ al final: PC-VENDES-01$
  • ⚠️ Compte trencat: si el canal segur es perd (p.ex. es restaura una imatge antiga), el PC no pot autenticar-se al domini. Solució: Test-ComputerSecureChannel -Repair

Perfils mòbils, obligatoris i carpetes personals

Centralitzar la configuració i les dades dels usuaris al servidor per garantir la mobilitat i la seguretat.

Tema 7 — Perfils

Tipus de perfils d'usuari

El perfil d'un usuari és el conjunt de configuració personal (escriptori, documents, favorits, configuració d'aplicacions) que s'associa al seu compte. La decisió sobre quin tipus de perfil usar és clau en la gestió d'una infraestructura de domini.

🏠

Perfil Local

El perfil es crea i s'emmagatzema localment a cada equip on l'usuari inicia sessió. Ubicació: C:\Users\nom_usuari\

No mòbil Configuració per equip
Quan usar-lo

Equips d'ús personal fixe. Llocs de treball no compartits. Entorns petits sense domini. ❌ No recomanat en dominis grans.

☁️

Perfil Mòbil (Roaming)

El perfil s'emmagatzema al servidor (carpeta compartida) i es sincronitza en cada inici/tancament de sessió. L'usuari té el mateix entorn en qualsevol equip del domini.

Mòbil Sincronitzat
Configuració

A la propietat de l'usuari AD: Profile Path → \\SRV-FILE-01\Profiles$\%username%. Windows crea automàticament la subcarpeta.

🔒

Perfil Obligatori

Variant del perfil mòbil on els canvis no es guarden en tancar la sessió. L'usuari sempre arrenca amb la mateixa configuració base. Ideal per a quioscos, aules o llocs de treball públics.

Només lectura Centralitzat
Com crear-lo

Reanomenar el fitxer ntuser.dat a ntuser.man a la carpeta del perfil del servidor. El ".man" (mandatory) indica a Windows que no ha de guardar els canvis.

⚠️
Problema dels perfils mòbils en entorns moderns:

Els perfils mòbils clàssics sincronitzen tot el perfil (incloent AppData amb GBs de dades d'aplicació), causant inicis/tancaments de sessió molt lents. La solució moderna és usar Folder Redirection (redirecció de carpetes) per redirigir Documents, Escriptori i AppData al servidor, sense sincronitzar tot el perfil. Combinat amb Offline Files, l'usuari treballa en local i sincronitza amb el servidor en segon pla.

Tema 8 — Carpetes personals

Carpetes personals (Home Folders)

La carpeta personal (Home Folder) és una carpeta al servidor on l'usuari emmagatzema els seus fitxers de treball. És diferent del perfil (que és configuració). En Linux, és equivalent al directori /home/usuari.

🪟 Configuració de carpetes personals en AD

  1. Crear la carpeta compartida al servidor (p.ex. \\SRV-FILE-01\Homes$). El $ al final oculta el recurs de la navegació de xarxa.

  2. Assignar permisos de compartició: Domain Users → Control total (els permisos NTFS de la subcarpeta de cada usuari restringiran l'accés real).

  3. A ADUC o PowerShell, assignar el Home Folder: -HomeDirectory "\\SRV-FILE-01\Homes$\%username%" i -HomeDrive "H:"

  4. Windows crea automàticament la subcarpeta per a l'usuari i assigna el compte d'usuari com a propietari amb permisos exclusius.

  5. En iniciar sessió, l'usuari veurà la unitat H: (o la lletra configurada) com a carpeta personal. Aplicacions com Word i Excel la usaran per defecte.

🐧 Redirecció de carpetes + Offline Files (GPO)

La redirecció de carpetes via GPO és la forma moderna de gestionar les dades dels usuaris:

  • 📁 Redirigir "Documentos": \\SRV-FILE\Redirected\%username%\Documents
  • 🖥️ Redirigir "Escritorio": \\SRV-FILE\Redirected\%username%\Desktop
  • 📸 Redirigir "Imágenes": permet backups centralitzats de totes les dades de l'usuari.
  • 🔌 Offline Files: les carpetes redirigides es sincronian a la caché local. L'usuari pot treballar sense xarxa (portàtils).
GPO — On configurar-ho

Configuració d'usuari → Configuració de Windows → Redirecció de carpetes. Per a Offline Files: Configuració d'equip → Configuració de Windows → Fitxers fora de línia.

Crear i usar plantilles d'usuari

Administració de grups, tipus i estratègies

Grups de seguretat i distribució, àmbits (local, global, universal), grups predeterminats i estratègia AGDLP.

Tipus de grups

Tipus i àmbits de grups a Active Directory

A Active Directory, els grups es classifiquen per tipus (per a quin propòsit s'usen) i per àmbit (on i a qui poden contenir). Confondre'ls és el primer error que comet un admin novell.

📋 Tipus de grup

  • 🔐 Grup de seguretat (Security Group):
    S'usa per assignar permisos a recursos (carpetes, impressores, aplicacions). Té un SID. És la meitat del model AGDLP. El tipus que més s'usa en infraestructura.
  • 📧 Grup de distribució (Distribution Group):
    S'usa exclusivament per a llistes de distribució de correu (Exchange/Microsoft 365). No pot assignar permisos. No té SID. No confondre'l amb un grup de seguretat.

🌐 Àmbits de grup

🏢
Global (GG)
Agrupa usuaris d'un domini. S'usa per representar un rol o departament. Pot contenir: usuaris i GGs del mateix domini. Visible a tot el bosc.
🔒
Local de Domini (DL)
S'usa per assignar permisos a recursos d'un domini. Pot contenir: usuaris, GGs i UGs de qualsevol domini. Però els permisos que atorga es limiten al domini local.
🌍
Universal (UG)
Conté usuaris i grups de qualsevol domini del bosc. El contingut es replica al Global Catalog (GC). Útil en boscos multi-domini. Cost de replicació elevat: usar-lo amb moderació.
Estratègia AGDLP

L'estratègia d'imbricació AGDLP (i AGUDLP)

AGDLP (Account → Global Group → Domain Local Group → Permission) és la best practice de Microsoft per gestionar permisos de manera escalable, ordenada i fàcil de mantenir en entorns de domini.

📐 Com funciona AGDLP pas a pas

👤
A — Account (Compte d'usuari o d'equip)
P.ex.: jgarcia, apuig, PC-VENDES-01$ (usuaris del domini)
→
👥
G — Global Group (Grup Global, per rol/departament)
P.ex.: Vendes_GG, IT_GG, RRHH_GG — agrupa els comptes d'usuaris pel seu rol
→
🔒
DL — Domain Local Group (Grup Local, per recurs)
P.ex.: RecursosCarpeta_Vendes_Lectura_GL, Impressora_Color_GL — un grup per a cada recurs i nivell d'accés
→
📁
P — Permission (Permís assignat al recurs)
El DL rep els permisos NTFS (Lectura, Modificació, Control total) sobre la carpeta, impressora o aplicació
🎯
Per quin motiu AGDLP és millor que assignar permisos directament a l'usuari?

❌ Sense AGDLP (mal pràctica):
Usuari jgarcia té permís directe a 47 carpetes de xarxa. Quan marxa de l'empresa, cal revisar les 47 carpetes per retirar-li el permís. El risc d'oblidar-ne una és molt alt.

✅ Amb AGDLP:
Usuari jgarcia és membre de Vendes_GG. Quan marxa, es treu del grup. Automàticament perd l'accés a tots els recursos als quals Vendes_GG donava accés. Una sola acció, efecte global.

# Verificar permisos NTFS resultants
PS C:\> (Get-Acl "C:\Recursos\Vendes").Access | Format-Table IdentityReference, FileSystemRights, AccessControlType

El que heu après

Els 10 continguts del RA4 i les seves aplicacions pràctiques com a tècnics ASIX administrant un domini Windows.

🏗️

Client/Servidor (T1)

Avantatges del domini vs workgroup. Rols del servidor (DC, DNS, DHCP, File, CA). Prerequisits per crear un domini. Install-ADDSForest, Get-ADDomain.

🌳

LDAP (T2)

Ports 389/636. Estructura DIT, DN, OU, CN, DC. Atributs clau. Format LDIF. ldapsearch, Get-ADUser -Properties *, DistinguishedName.

🏢

Domini & OUs (T3)

Bosc, arbre, domini. Estratègies de disseny d'OUs (geogràfica, per departament, híbrida). 5 rols FSMO. New-ADOrganizationalUnit, Get-ADForest.

👤

Comptes (T4–6)

Comptes predeterminades (RID-500/501/502). SID vs SamAccountName. PSO Fine-Grained. Unir equips. New-ADUser, New-ADFineGrainedPasswordPolicy, Add-Computer.

🔒

Seguretat (T5)

Política de contrasenyes (longitud, complexitat, historial). Bloqueig de comptes (llindar, finestra, durada). Account Lockout DoS. krbtgt.

☁️

Perfils & Carpetes (T7–8)

Local vs mòbil vs obligatori (ntuser.man). Home Folder (H:). Folder Redirection GPO. Offline Files. Set-ADUser -ProfilePath -HomeDirectory -HomeDrive.

📋

Plantilles (T9)

Plantilles amb prefix _TPL-, desactivades. Atributs que es copien. Import CSV massiu. Variables %USERNAME%, %HOMEDRIVE%. Scripts de logon.

👥

Grups AGDLP (T10)

Seguretat vs distribució. Global (G), DomainLocal (DL), Universal (U). Estratègia AGDLP. Grups predeterminats (DA, EA, Schema Admins). New-ADGroup, Add-ADGroupMember.
