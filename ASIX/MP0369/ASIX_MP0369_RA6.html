Mètriques fonamentals

Les 4 mètriques clau d'un sistema

Qualsevol problema de rendiment pot traçar-se a un dels quatre recursos fonamentals del sistema. Un tècnic ASIX ha de saber llegir cadascun, identificar valors anòmals i correlacionar-los entre si.

🖥️

Panell de mètriques d'un servidor en producció

8%
RAM usada
6,8G/8G
Swap usada
2,2G
Disc I/O r
45MB/s

⚠️

Diagnosi: Swap activa (2,2 GB) i RAM al 85% → el sistema ha exhaurit la RAM física i fa ús del disc virtual → rendiment degradat. Investigar quin procés consumeix la RAM.

⚙️

CPU — Interpretar l'ús

  • 👤 user (us): temps en processos d'usuari. Alt és normal si hi ha càrrega legítima.
  • 🔧 system (sy): temps al kernel. >15% indica sobrecàrrega de crides al sistema.
  • ⏳ iowait (wa): CPU esperant I/O de disc. >20% indica coll d'ampolla de disc.
  • 🔴 steal (st): temps robat per l'hipervisor (en VMs). >5% indica contencions al host físic.
💾

RAM — La tríada: free, cached, buffer

Interpretació de free -h

total=8G, used=6.8G, free=0.2G, buff/cache=1.0G, available=1.2G
Clau: available (no free) és la RAM realment usable sense fer swapping. Si available és baix i Swap s'usa activament → problema greu.

Lab 1.1 — Monitoratge en temps real i identificació de colls d'ampolla

Genera càrrega artificial al sistema i identifica els recursos afectats amb les eines de temps real

🔀 Ambdós

Generareu càrrega artificial als recursos del sistema i aprendreu a llegir les mètriques per identificar el recurs saturat.

🐧 Ubuntu — Generació de càrrega i monitoratge
Generació de càrrega i monitoratge en paral·lel🐧 Ubuntu
# Instal·lar eines de generació de càrrega
$ sudo apt install stress-ng htop sysstat -y
# Terminal 1: monitorar en temps real
$ htop
# Terminal 2: generar càrrega CPU (4 workers durant 60s)
$ stress-ng --cpu 4 --timeout 60s --metrics-brief
# Observa a htop: CPU puja al ~100% per cada nucli
# Generar càrrega RAM (800MB)
$ stress-ng --vm 1 --vm-bytes 800M --timeout 30s
# Generar càrrega I/O disc
$ stress-ng --io 4 --timeout 30s
# A htop, observa iowait (wa) pujant
# Monitorar amb watch (actualitza cada 2s)
$ watch -n 2 "free -h && echo '---' && iostat -xz 1 1 | tail -5"
# Comprovar load average contínuament
$ watch -n 1 "uptime && cat /proc/loadavg"
🪟 Windows — Generació de càrrega
PowerShell — Generació de càrrega CPU i RAM🪟 Windows
# Càrrega CPU (usa tots els nuclis durant 30s)
PS> 1..4 | ForEach-Object { Start-Job { while ($true) { [math]::Sqrt(1234567) } } }
PS> Start-Sleep 30; Get-Job | Stop-Job; Get-Job | Remove-Job
# Monitorar CPU en temps real (cada 2s, 30 mostres)
PS> Get-Counter "\Processor(_Total)\% Processor Time" -SampleInterval 2 -MaxSamples 30 | Select-Object -ExpandProperty CounterSamples | Select CookedValue
# Monitorar RAM disponible
PS> Get-Counter "\Memory\Available MBytes" -SampleInterval 2 -MaxSamples 15
  • Documenta la línia base (baseline): Sense càrrega artificial, anota els valors normals de CPU, RAM, disc i xarxa. Aquests valors de referència et permetran detectar anomalies futures.
  • Experimenta amb cada tipus de càrrega (CPU, RAM, I/O) per separat. Documenta quin indicador a htop/Task Manager és el primer en reaccionar per a cada tipus de càrrega.
  • Interpreta el Load Average: en un equip de 4 nuclis, si el load average és 4.0, 6.0 o 0.5, quin indica sobrecàrrega? Explica el concepte de "load average normalitzat" (load / nombre de nuclis).
  • Identifica un procés zombie: Crea un procés zombie artificialment i comprova com apareix a htop (estat Z). bash -c '(sleep 5 & wait; kill -9 $$) &'
📋 Documenta: Captura de htop/Task Manager amb cada tipus de càrrega activa. Taula de valors baseline vs sota càrrega. Explicació de quin indicador es va saturar primer i per quin motiu.

Eines de monitoratge continu

Prometheus + Grafana, Zabbix, Nagios, sar, collectd. Dashboards, alertes automàtiques i tendències.

Arquitectura de monitoratge empresarial

De les mètriques puntuals al monitoratge continu: per quin motiu?

Les eines de temps real (top, htop) mostren l'estat actual però no conserven l'historial. Un problema de rendiment que passa a les 3h de la matinada és indetectable sense monitoratge continu. Els sistemes professionals recullen, emmagatzemen i visualitzen mètriques durant mesos o anys.

🎯

Per a què serveix el monitoratge continu?

  • 📉 Detectar tendències: un disc que s'omple gradualment, una fuga de memòria que creix each dia.
  • ⚠️ Alertes proactives: avisar ABANS que el servei caigui. Ex: alerta quan disc > 80%.
  • 🔎 Post-mortem: analitzar l'historial per entendre per quin motiu va caure un servei ahir.
  • 📊 Planificació: predir quan caldrà ampliar el hardware basant-se en tendències reals.
🔧

sar — System Activity Reporter

Part del paquet sysstat. Recull mètriques cada 10 min (configurable) i les guarda a /var/log/sysstat/. Gratuït, lleuger, sempre disponible.

sar — exemples🐧
$ sudo apt install sysstat -y
# Activar recopilació automàtica
$ sudo sed -i 's/ENABLED="false"/ENABLED="true"/' /etc/default/sysstat
$ sudo systemctl restart sysstat
# CPU dels últims 7 dies
$ sar -u -f /var/log/sysstat/sa$(date +%d -d yesterday)
# RAM d'avui (cada 10min)
$ sar -r 1
# Disc I/O d'avui
$ sar -b 1
# Xarxa d'avui
$ sar -n DEV 1

Prometheus + Grafana

Prometheus + Grafana: l'stack de monitoratge modern

Prometheus és una base de dades de sèries temporals que recull mètriques fent scraping (pull) a endpoints HTTP cada N segons. Grafana és la capa de visualització que crea dashboards interactius sobre les dades de Prometheus.

Instal·lació Prometheus + Node Exporter + Grafana🐧 Ubuntu
# Node Exporter: exposa mètriques del sistema a /metrics
$ wget https://github.com/prometheus/node_exporter/releases/latest/download/node_exporter-*.linux-amd64.tar.gz
$ tar xvf node_exporter-*.tar.gz && sudo mv node_exporter-*/node_exporter /usr/local/bin/
$ sudo systemctl enable --now node_exporter
# Verificar que exposa mètriques
$ curl http://localhost:9100/metrics | head -20
node_cpu_seconds_total{cpu="0",mode="idle"} 12345.67
node_memory_MemAvailable_bytes 1.234e+09
node_filesystem_avail_bytes{mountpoint="/"} 4.567e+10
# Prometheus: recull les mètriques del node_exporter
# Fitxer /etc/prometheus/prometheus.yml
scrape_configs:
- job_name: 'node'
scrape_interval: 15s
static_configs:
- targets: ['localhost:9100', '192.168.1.20:9100']
# Alertes a alertmanager.yml (alerta quan disc > 80%)
- alert: DiskAlmostFull
expr: node_filesystem_avail_bytes / node_filesystem_size_bytes * 100 < 20
for: 5m
annotations:
summary: "Disc gairebé ple: {{ $labels.instance }}"
# Query PromQL: CPU % per host (últims 5min)
100 - (avg by (instance)(rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

Zabbix — Monitoratge empresarial tradicional

Zabbix és un sistema de monitoratge madur amb agent, agentless (SNMP), checks actius/passius, escalada d'alertes i informes. Molt usat en grans empreses.

  • 🔧 Agent: instal·la zabbix-agent2 al servidor a monitorar. Comunica amb el server central.
  • 📋 Templates: plantilles preconfigurades per a Linux, Windows, MySQL, Apache, etc.
  • 🔔 Alertes: correu, Slack, Telegram, webhook quan un trigger dispara.
Instal·lar agent Zabbix🐧
$ sudo apt install zabbix-agent2 -y
# /etc/zabbix/zabbix_agent2.conf
Server=192.168.1.5 # IP del Zabbix Server
Hostname=srv-web-01
$ sudo systemctl enable --now zabbix-agent2

Anàlisi de subsistemes

Eines de diagnosi profunda per subsistema

Quan l'eina de temps real indica un problema, cal eines d'anàlisi per identificar la causa arrel. Cada subsistema (CPU, memòria, disc, xarxa) té les seves eines especialitzades.

Anàlisi de rendiment per subsistema — Ubuntu i Windows🔀 Ambdós
══════ CPU — perf i mpstat ══════
$ mpstat -P ALL 2 5 # CPU per nucli, 5 mostres cada 2s
07:30:01 CPU %usr %sys %iowait %steal %idle

07:30:03 0 91.00 9.00 0.00 0.00 0.00 ← Nucli 0 saturat!
07:30:03 1 35.00 4.00 18.00 0.00 43.00

$ sudo perf top -p 1842 # Top de funcions del PID 1842 en temps real
$ sudo perf record -g -p 1842 sleep 10 && sudo perf report

$ vmstat 2 10 # Estadístiques de VM cada 2s, 10 vegades

Diagnosi avançada: strace i lsof

strace i lsof: seguir la pista d'un procés fins a la causa arrel

🕵️

strace — interceptar crides al sistema

strace intercepta totes les crides al sistema (syscalls) que fa un procés. Permet veure quins fitxers obre, quines connexions fa i on es queda penjat.

strace — exemples de diagnosi🐧
# Seguir un procés existent (PID 1842)
$ sudo strace -p 1842 -e trace=network
connect(5, {sa_family=AF_INET, sin_port=3306...} ← Connexió a MySQL
# Crides al sistema d'un programa (quins fitxers obre)
$ strace -e trace=open,openat ls /etc 2>&1 | head -10
# Temps per crida (on es perd temps)
$ sudo strace -p 1842 -T -e trace=read,write 2>&1 | head -20
📂

lsof — Llistar fitxers oberts

lsof (List Open Files) mostra tots els fitxers, sockets i pipes oberts per qualsevol procés. A Linux, tot és un fitxer.

lsof — diagnosi de fitxers i xarxa🐧
# Qui té obert un fitxer específic?
$ sudo lsof /var/log/syslog
# Fitxers oberts per un procés
$ sudo lsof -p 1842
# Connexions de xarxa actives d'un procés
$ sudo lsof -p 1842 -i
# Quins processos escolten al port 443?
$ sudo lsof -i :443
# Fitxers esborrats però encara en ús (espai alliberat)
$ sudo lsof | grep deleted | awk '{print $1,$2,$7}' | sort -k3 -rn | head

Lab 3.1 — Diagnosi professional: identifica la causa arrel d'un problema de rendiment

Investiga un escenari de problema real: servidor lent, disc saturat, fuga de memòria

🔀 Ambdós

Simularàs un servidor web amb problemes i aplicaràs el mètode d'investigació sistemàtic: Observar → Mesurar → Hipòtesi → Verificar → Solucionar.

  • Escenari 1 — Disc saturat: Genera un fitxer enorme: dd if=/dev/zero of=/tmp/big.dat bs=1M count=5000 mentre monitores amb iostat -xz 2. Observa el %util del disc. Usa sudo iotop -o per identificar el procés culpable. Calcula quan trigaria a omplir el disc al ritme actual (df -h + velocitat d'escriptura).
  • Escenari 2 — Fuga de memòria simulada: Executa: python3 -c "data=[]; [(data.append(' '*1024*1024)) for _ in range(500)]". Mentre s'executa, monitora amb watch -n 1 "ps -o pid,rss,comm -C python3". Observa com creix el RSS. Usa vmstat 1 per veure l'impacte al swap.
  • Escenari 3 — Connexions de xarxa esgotades: Instal·la Apache (sudo apt install apache2). Usa ab -n 10000 -c 500 http://localhost/ per generar càrrega. Monitora: ss -s (estadística de sockets), netstat -an | grep TIME_WAIT | wc -l. Interpreta l'estat TIME_WAIT.
  • Windows — Performance Monitor: Crea un Data Collector Set que enregistri cada 5s durant 5 min: % de temps del processador, MB disponibles, Pàgines/s, Temps d'inactivitat del disc. Analitza el fitxer .blg generat i identifica el pic de càrrega.
📋 Documenta: Per a cada escenari: mètriques observades, hipòtesi inicial, confirmació, causa arrel identificada i solució proposada. Format: informe d'incident amb timestamp, símptomes, diagnosi i resolució.

Registre de successos — Logs del sistema

journald, /var/log, syslog, Visor d'Esdeveniments Windows. Lectura, filtratge i anàlisi de logs.

Arquitectura de logs a Linux

journald i /var/log: els dos sistemes de logs de Linux

Linux modern usa journald (part de systemd) com a sistema de logging principal: recopila logs de tots els serveis en format binari indexat. Els logs clàssics en text pla segueixen disponibles a /var/log/, alimentats per rsyslog/syslog-ng.

📖

journalctl — La porta al journald

journalctl — referències completes🐧 Ubuntu
# Logs de tot el sistema (des de l'arrencada)
$ journalctl
# Seguir en temps real (com tail -f)
$ journalctl -f
# Logs d'un servei concret
$ journalctl -u nginx.service -f
$ journalctl -u ssh.service --since "1 hour ago"
# Filtrar per prioritat (err,warning,info,debug)
$ journalctl -p err -b # Errors de l'arrencada actual (-b=boot)
$ journalctl -p warning --since today
# Filtrar per rang de temps
$ journalctl --since "2025-06-01 08:00" --until "2025-06-01 10:00"
# Format JSON (per processar amb jq)
$ journalctl -u sshd -o json-pretty | jq '.MESSAGE'
# Veure arrencades anteriors
$ journalctl --list-boots
$ journalctl -b -1 -p err # Errors de l'arrencada anterior
# Gestió d'espai del journal
$ journalctl --disk-usage
$ sudo journalctl --vacuum-size=500M # Netejar fins a 500MB

Windows — Visor d'Esdeveniments

Event Viewer (eventvwr) i PowerShell: els logs de Windows

Windows centralitza tots els logs al Visor d'Esdeveniments (Event Viewer). Cada log és un arxiu .evtx. Les entrades es classifiquen per nivell de gravetat i es filtren per criteris XPath.

Nivells de gravetat dels events

Critical (1)Error greu que pot causar pèrdua de dades. Ex: Kernel panic, BSOD, fallada hardware.
Error (2)Funcionalitat fallida però el sistema segueix. Ex: Servei que no arrenca, driver que falla.
Warning (3)Possible problema futur. Ex: Disc al 85%, disc defectuós (S.M.A.R.T.), timeout de xarxa.
Information (4)Events normals del sistema. Ex: Servei iniciat, usuari ha iniciat sessió, instal·lació de software.
Verbose (5)Informació de depuració molt detallada. Desactivat per defecte (ompliria el disc).
PowerShell — Get-EventLog i Get-WinEvent🪟 Windows
# Logs disponibles al sistema
PS> Get-WinEvent -ListLog * | Where-Object {$_.RecordCount -gt 0} | Select LogName,RecordCount
# Errors del Sistema de les últimes 24h
PS> Get-WinEvent -FilterHashtable @{LogName='System';Level=2;StartTime=(Get-Date).AddHours(-24)} | Select TimeCreated,Id,Message | Format-Table -Wrap

Tot procés a Linux té un PID (Process ID) i un PPID (Parent Process ID). Els processos s'organitzen en un arbre jeràrquic on systemd (PID 1) és l'arrel. Entendre l'arbre és fonamental per gestionar processos correctament.

cgroups i systemd — Control de recursos

cgroups: limitar els recursos d'un procés o servei

Els Control Groups (cgroups) permeten limitar, prioritzar i aïllar els recursos (CPU, RAM, I/O, xarxa) que un grup de processos pot consumir. systemd usa cgroups per a cada servei de forma transparent.

cgroups i systemd — control de recursos per servei🐧 Ubuntu
# Veure l'ús de recursos per servei (systemd + cgroups)
$ systemd-cgtop # Com top però per cgroup/servei
$ systemctl status mysql # Inclou: Memory: 1.4G, CPU: 38%
# Limitar CPU i RAM d'un servei via unit file
$ sudo systemctl edit nginx # Crea override.conf
# Contingut de /etc/systemd/system/nginx.service.d/override.conf:
[Service]
CPUQuota=50% # Màxim 50% de CPU (1 nucli en un sistema de 2)
MemoryMax=512M # Màxim 512MB de RAM. Si supera → OOM killer
MemoryHigh=400M # A partir de 400MB comença a fer backpressure
IOWeight=50 # Pes d'I/O (100=normal, 50=meitat de prioritat)
$ sudo systemctl daemon-reload && sudo systemctl restart nginx
# Verificar límits aplicats
$ systemctl show nginx | grep -E "CPU|Memory|IO"
# Veure jerarquia cgroups
$ cat /sys/fs/cgroup/system.slice/nginx.service/memory.current
$ cat /sys/fs/cgroup/system.slice/nginx.service/cpu.stat
# OOM Score: quin procés mata el kernel quan la RAM s'esgota
$ cat /proc/1842/oom_score # Valor alt = candidat a ser matat primer
$ echo -1000 | sudo tee /proc/1842/oom_score_adj # Protegir MySQL del OOM killer
Processos especials

Processos zombies, orfes i dimonis

🧟

Processos zombie (Z)

Un procés zombie ha acabat però el seu pare no ha recollit el codi de sortida (wait()). Ocupa una entrada a la taula de processos però no consumeix CPU ni RAM.

Diagnosi

ps aux | grep Z per detectar-los. ps -eo ppid,pid,stat,comm | grep Z per identificar el pare. La solució és matar el procés PARE (no el zombie).

⚠️

Un sistema amb centenars de zombies indica un bug al procés pare. Pot exhaurir la taula de PIDs (/proc/sys/kernel/pid_max).

👻

Processos orfes

Un procés orfe el pare del qual ha mort. A Linux, init/systemd (PID 1) adopta automàticament els processos orfes. No sol ser un problema.

Exemple

Un administrador llança un procés llarg per SSH i tanca la sessió. Si el procés estava en primer pla, mor; si estava en background desvinculat (nohup, screen, tmux), continua com a orfe adoptat per systemd.

🔮

Dimonis (daemons)

Processos de llarga durada que s'executen en segon pla, sense terminal associat. Normalment el nom acaba en d: sshd, mysqld, nginx (worker).

  • 🔧 systemd: gestiona tots els dimonis al Linux modern via units (.service).
  • 🪟 Windows Services: equivalent Windows. Gestió amb services.msc o PowerShell.

El que heu après

Repàs dels conceptes clau del RA6 i la seva aplicació professional com a tècnics ASIX.

⚡

Temps Real

Linux: top (us/sy/wa/st, load avg), htop, btop, glances. Windows: Task Manager (Ctrl+Shift+Esc), resmon, perfmon. Clau: iowait alt = disc, si/so = swap actiu, load/nuclis = saturació.

📈

Monit. Continu

Stack: Prometheus (pull, PromQL) + Grafana (dashboards) + Alertmanager. Alternatives: Zabbix (agent-based), Netdata (auto). CLI: sar -u/-r/-b/-n. Alertes: disc >80%, RAM <512MB.

📋

Logs

Linux: journalctl (-f, -u, -p, -b, --since). /var/log/auth.log (SSH). logrotate. Windows: eventvwr, Get-WinEvent, IDs: 4625 (login fallit), 4624 (ok), 4720 (compte creat), 41 (apagada inesperada).

⚙️

Processos

ps, kill, nice/renice. Signals: SIGTERM (15, graceful), SIGKILL (9, força). cgroups: CPUQuota, MemoryMax. Zombie: fill mort, pare no fa wait(). OOM Killer. Windows: Get-Process, Stop-Process, prioritats.
