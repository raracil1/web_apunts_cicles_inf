Interpretació, anàlisi i elaboració de documentació tècnica

Estàndards de documentació, eines, tipus de documents i com llegir fitxers tècnics (man pages, RFC, datasheets, changelogs).

Per què la documentació importa

La documentació com a actiu crític de TI

En TI professional, un sistema sense documentació és un sistema de caixa negra: quan falla (i fallarà), ningú sap com funciona per arreglar-lo. La documentació tècnica és el conjunt de documents que descriuen l'arquitectura, la configuració, els procediments i el manteniment dels sistemes. No és opcional: és part del servei que es lliura.

📐

Tipus de documentació tècnica

  • 🗺️ Diagrames d'arquitectura: topologia de xarxa, diagrama de servidors, fluxos de dades. Eines: Lucidchart, draw.io, Visio.
  • 📋 Inventari d'actius (CMDB): llista de tots els equips, IPs, sistemes operatius, software, responsable. Eines: GLPI, OCS Inventory, Snipe-IT.
  • 📖 Runbooks / SOP: procediments pas a pas per a tasques recurrents. "Com reiniciar el servidor de correu", "Com fer un backup manual".
  • 🔧 Documentació de configuració: fitxers de configuració comentats, valors usats, raons per les decisions.
  • 📊 Changelog / Release Notes: registre de canvis aplicats al sistema (qui, quan, què, per quin motiu).
📚

Fonts de documentació tècnica externes

  • 📄 Man pages (man): referència definitiva de cada comanda a Linux. Sempre llegir la secció SYNOPSIS, OPTIONS i EXAMPLES.
  • 📡 RFC (Request for Comments): estàndards de protocols d'Internet (IETF). Ex: RFC 7321 = DNS, RFC 5321 = SMTP, RFC 8446 = TLS 1.3.
  • 📰 KB Microsoft (Knowledge Base): articles tècnics de Microsoft sobre errors, configuració i solucions. Format KB + número.
  • 🐧 Ubuntu Server Guide / man.openbsd.org: documentació oficial de les distribucions.
  • 🔖 CVE / NVD: National Vulnerability Database. Cada CVE té descripció tècnica, puntuació CVSS i mitigació.
📏

Estàndards de qualitat documental

  • ✍️ Claredat: qualsevol tècnic del teu nivell hauria de poder seguir el document sense preguntar.
  • 🔄 Actualització: un document obsolet és pitjor que cap document. Incloure data de revisió i responsable.
  • 🔢 Versionat: usar control de versions (Git, Confluence, SharePoint) per poder veure l'historial de canvis.
  • 🔍 Cerca: usar etiquetes, índexs i estructura coherent per trobar-ho ràpidament en un incident.
  • 🔐 Confidencialitat: la documentació amb contrasenyes, IPs internes o arquitectura s'ha de protegir i xifrar.
🧠
Eines de documentació per a tècnics ASIX:
Confluence (wiki empresarial) Notion / Obsidian draw.io / Lucidchart Git (versionat docs) MkDocs / Docusaurus GLPI (inventari + ticketing) SharePoint Markdown (.md)

🔀 Windows + Ubuntu

Documentarem automàticament la configuració del sistema per generar la base d'una CMDB (Configuration Management Database).

🐧 Ubuntu — Script de documentació automàtica del sistema
Terminal — Script de documentacióUbuntu
# Script que genera un informe de configuració del sistema
root@host:~# cat > /usr/local/bin/doc-sistema.sh << 'EOF'
#!/bin/bash
OUTPUT="/tmp/doc-sistema-$(hostname)-$(date +%Y%m%d).md"
echo "# Documentació del Sistema: $(hostname)" > $OUTPUT
echo "**Data:** $(date)" >> $OUTPUT
echo "## Sistema Operatiu" >> $OUTPUT
lsb_release -a 2>/dev/null >> $OUTPUT
uname -a >> $OUTPUT
echo "## Hardware" >> $OUTPUT
echo "### CPU: $(lscpu | grep 'Model name' | cut -d: -f2 | xargs)" >> $OUTPUT
echo "### RAM: $(free -h | awk '/^Mem/{print $2}')" >> $OUTPUT
echo "## Xarxa" >> $OUTPUT
ip addr show >> $OUTPUT
echo "## Disc i particions" >> $OUTPUT
lsblk -f >> $OUTPUT
df -hT >> $OUTPUT
echo "## Serveis actius" >> $OUTPUT
systemctl list-units --type=service --state=active >> $OUTPUT
echo "## Paquets instal·lats" >> $OUTPUT
dpkg --get-selections | wc -l >> $OUTPUT
echo "Informe generat a: $OUTPUT"
EOF
root@host:~# chmod +x /usr/local/bin/doc-sistema.sh && doc-sistema.sh

Manuals d'instal·lació i configuració

Com redactar, estructurar i mantenir manuals tècnics per a sistemes operatius i aplicacions d'empresa.

Estructura d'un manual tècnic

Anatomia d'un manual d'instal·lació professional

Un manual d'instal·lació ben escrit permet a un tècnic replicar la configuració d'un sistema sense cap coneixement previ i sense haver de preguntar. La clau és l'especificitat: cada pas ha d'indicar el comando exacte, el valor a usar i el resultat esperat.

📋 Seccions obligatòries d'un manual d'instal·lació

  1. Objectiu i abast: Per a quin servei és, en quins sistemes s'aplica, qui és el destinatari (admin novell vs expert).

  2. Prerequisits: Maquinari mínim, SO requerit, ports que cal tenir oberts, credencials necessàries, dependències prèvies.

  3. Procediment d'instal·lació: Cada pas numerado amb el comand exacte, els valors a introduir i una captura si cal.

  4. Configuració post-instal·lació: Fitxers de configuració a modificar, paràmetres específics de l'entorn (IPs, noms de domini, contrasenyes).

  5. Verificació: Com comprovar que la instal·lació ha anat bé. Comandos de test, ports oberts, output esperat.

  6. Solució de problemes freqüents: Errors habituals, les seves causes i solucions. Arxius de log a consultar.

  7. Desinstal·lació / Rollback: Com desfer la instal·lació si cal. Imprescindible per a entorns de producció.

✅ Bones pràctiques de redacció

  • 🎯 Usa veu activa i imperativa: "Executa", "Edita", "Comprova". No "S'ha d'executar".
  • 💻 Distingueix comandos del text: usa blocs de codi (monospace) per als comandos, sortides i noms de fitxers.
  • ⚠️ Marca les zones de risc: "⚠️ Atenció: aquest pas esborra tot el disc", "🔐 Substitueix CONTRASENYA per la real".
  • 🔢 Usa variables per als valors que canvien: IP_SERVIDOR, NOM_DOMINI, VERSIO.
  • 🧪 Prova el manual des de zero: un manual que no s'ha provat no és fiable. Fes-lo seguir a algú que no sap el sistema.

Llicències de client i llicències de servidor

CAL (Client Access License), llicències per núcleo, OEM, Volume Licensing, SPLA i llicències de programari lliure en entorns empresarials.

Model de llicències Microsoft

Llicències de servidor i d'accés de client (CAL)

El model de llicències de Microsoft per a productes de servidor és complex. Entendre'l és imprescindible per a qualsevol tècnic ASIX que hagi d'implantar o auditar sistemes Windows en una empresa. Adquirir incorrectament les llicències pot resultar en multes i costos inesperats.

🖥️

👤

CAL — Client Access License

Per accedir a la majoria de productes de servidor Microsoft (Windows Server, Exchange, SQL Server, SharePoint), cada client necessita una CAL addicional a la llicència del servidor.

Tipus CALAssignada aQuan usar
User CALUn compte d'usuariL'usuari accedeix des de múltiples dispositius (portàtil, mòbil, PC fixe)
Device CALUn dispositiuVaris usuaris comparteixen el mateix PC (torns de treball, quioscos)
RDS CALUsuari/DispositiuAccés a Remote Desktop Services (escriptori remot / Terminal Server)
External ConnectorEl servidor (il·limitat)Usuaris externs (clients, proveïdors) accedeixen al servidor
⚠️
Trampes habituals en llicències Microsoft:
  • La llicència del SO no inclou les CALs: comprar Windows Server 2022 Standard NO inclou les CALs per als clients que s'hi connecten. Cal comprar-les a part.
  • Virtualització i Standard: Windows Server Standard llicencia 2 VMs. Per a 4 VMs calen 2 llicències Standard (no 1 Datacenter).
  • SQL Server té el seu propi model: llicències per core o per servidor+CAL, independentment de Windows Server.
  • OEM no és transferible: la llicència OEM queda lligada al hardware original. No es pot moure a un nou servidor.
Models de llicenciament alternatiu
📦

Volume Licensing (VL)

Per a empreses amb moltes llicències. Programes: Open License, EA (Enterprise Agreement), CSP (Cloud Solution Provider). Avantatges: descomptes per volum, gestió centralitzada al VLSC, Software Assurance (dret d'actualització).

☁️

SPLA — Service Provider License Agreement

Model de subscripció mensual per a proveïdors de serveis (hosters, MSPs) que volen oferir serveis Microsoft als seus clients. Es paga per usuari/mes actiu. Ideal per a empreses que ofereixen VDI, RDS o SaaS basat en Microsoft.

🐧

Llicències en programari lliure

El programari lliure té les seves pròpies consideracions: Ubuntu LTS és gratuït però Canonical ofereix suport comercial (Ubuntu Pro). RHEL és de pagament per subscripció però FOSS (el codi és obert). AlmaLinux/Rocky Linux = RHEL sense suport comercial.

4–5

Instal·lacions desateses i fitxers de resposta

Automatitzar la instal·lació de sistemes operatius a escala: autounattend.xml (Windows), preseed/cloud-init (Linux) i Windows Deployment Services (WDS).

Per quin motiu automatitzar?

La necessitat d'instal·lació desatesa en entorns d'empresa

Instal·lar manualment un sistema operatiu en un equip tarda entre 30-60 minuts amb la configuració bàsica. Multiplicat per 50 equips = 50 hores de treball manual i 50 oportunitats d'error. Les instal·lacions desateses permeten desplegar sistemes configurats de manera estàndard en minuts, sense intervenció humana.

🪟

autounattend.xml — Windows

L'instal·lador de Windows llegeix el fitxer autounattend.xml (o unattend.xml) durant la instal·lació per respondre automàticament a totes les preguntes. S'ubica a l'arrel del USB d'instal·lació o en un recurs de xarxa.

Eina: Windows SIM (WAIK/ADK) Generador web: schneegans.de
Passes del procés d'instal·lació (7 passes)

windowsPE: particionament i còpia de fitxers
offlineServicing: paquets i controladors offline
generalize: preparació per a imatge (sysprep)
specialize: hostname, domèstic, zona horària
auditSystem/auditUser: personalitzacions d'auditoria
oobeSystem: compte d'usuari, contrasenya, privacitat

🐧

preseed / cloud-init — Linux

Linux disposa de diverses alternatives per a instal·lació desatesa:

  • 📋 Preseed (Debian/Ubuntu Legacy): fitxer de text que respon automàticament a les preguntes de debconf. Funciona amb PXE o CD.
  • ☁️ cloud-init: estàndard de facto per a VMs i cloud (AWS, Azure, OpenStack). Fitxer YAML que configura l'equip en el primer arrencada.
  • 📝 Autoinstall (Ubuntu 20.04+): el format modern d'Ubuntu, en YAML. Substitueix preseed per als instal·ladors nous.
  • 🎩 Kickstart (RHEL/CentOS/Fedora): fitxer .ks que configura particions, paquets, usuaris i scripts post-instal·lació.
Fitxer autounattend.xml

Exemple d'autounattend.xml per a Windows 11

<unattend xmlns="urn:schemas-microsoft-com:unattend">

<!-- PASSA 1: windowsPE — Particionament i selecció d'idioma -->
<settings pass="windowsPE">
<component name="Microsoft-Windows-International-Core-WinPE">
<SetupUILanguage><UILanguage>ca-ES</UILanguage></SetupUILanguage>
<InputLocale>0403:0000040A</InputLocale>
<UILanguage>ca-ES</UILanguage>
<UserLocale>ca-ES</UserLocale>
</component>
<component name="Microsoft-Windows-Setup">
<!-- Acceptar la llicència automàticament -->
<UserData><AcceptEula>true</AcceptEula></UserData>
<!-- Particionament automàtic per a UEFI+GPT -->
<DiskConfiguration><Disk wcm:action="add">
<DiskID>0</DiskID>
<WillWipeDisk>true</WillWipeDisk>
<!-- Partition 1: EFI, 100MB -->
<!-- Partition 2: MSR, 16MB -->
<!-- Partition 3: Windows, resta del disc -->
</Disk></DiskConfiguration>
</component>
</settings>

<!-- PASSA 4: specialize — Configuració del sistema -->

<TimeZone>Romance Standard Time</TimeZone>

</settings>
</unattend>

Servidors d'actualitzacions automàtiques

WSUS (Windows Server Update Services) per a Windows i apt-cacher-ng/apt-mirror per a Ubuntu. Centralitzar i controlar les actualitzacions de tota la flota.

Per quin motiu un servidor d'actualitzacions?

La necessitat d'actualitzacions centralitzades

En una empresa amb 100 equips, si cada un descarrega les actualitzacions directament d'Internet, es consumeix ample de banda enorme. A més, no hi ha control sobre quines actualitzacions s'apliquen ni quan. Un servidor d'actualitzacions centralitzat resol ambdós problemes: estalvi d'ample de banda + control de la implementació.

🔄

WSUS — Windows Server Update Services

Rol de Windows Server que actua de proxy per a Microsoft Update. Descarrega les actualitzacions una vegada i les distribueix a tots els clients.

  • ⬇️ Sincronització: el servidor WSUS es sincronitza amb Microsoft Update periòdicament (un cop al dia). Els clients les descarreguen del servidor intern.
  • ✅ Aprovació manual: un admin aprova cada actualització abans que s'instal·li als clients. Permet provar en una màquina pilot primer.
  • 📊 Reporting: veure quins equips estan al dia, quins tenen actualizaciones pendents, quins han fallat.
  • 🔗 Integració amb GPO: els clients es configuren via GPO per apuntar al servidor WSUS en lloc de Microsoft Update.
🐧

apt-cacher-ng i apt-mirror — Ubuntu

Dos enfocaments per a centralitzar les actualitzacions APT a Linux:

apt-cacher-ng — Proxy caché (recomanat)

Actua com a proxy transparent: la primera vegada que un client demana un paquet, el descarrega d'Internet i el guarda en caché. Les properes vegades el serveix de la caché local. Sense necessitat de mirall complet. Molt menys espai en disc.

apt-mirror — Mirall complet (per a entorns offline)

Descarrega el repositori complet d'Ubuntu al servidor local. Útil en entorns sense accés a Internet. Requereix molt espai en disc (Ubuntu 24.04 complet: ~400 GB). Els clients apunten al servidor local com si fos un repositori d'Ubuntu oficial.

Tema 8 — Protocols d'actuació

Protocols d'actuació davant incidents habituals

Un protocol d'actuació és un procediment estandarditzat per respondre a un tipus d'incident concret. Tenir-los predefinits redueix el temps de resolució i garanteix que tots els tècnics actuïn de manera consistent.

🔐 Protocol: Compte bloquejada

  1. Verificar que és un bloqueig legítim (no un atac). Revisar EID 4740 al DC.

  2. Identificar l'equip origen del bloqueig. Comprovar si és el propi PC de l'usuari o un altre equip/servei.

  3. Si és l'equip de l'usuari: comprovar si hi ha credencials antigues guardades (Control Panel → Credential Manager).

  4. Desbloquejar el compte (Unlock-ADAccount) i demanar al usuari que canviï la contrasenya.

  5. Documentar al tiquet: causa, PC origen, hora, resolució.

💾 Protocol: Servidor sense espai en disc

  1. Identificar la partició afectada i quin directori ocupa més espai (du -sh /* / Get-ChildItem).

  2. Netejar logs antics (/var/log/), fitxers temporals (/tmp/) i backups locals antics.

  3. Si el log és el problema: rotar logs manualment (logrotate -f) o archivar-los.

  4. Ampliar disc si és necessari (LVM extend, Storage Spaces en Windows).

  5. Crear una alerta de monitoratge per avisar quan l'espai superi el 80% en el futur.

🔁 Protocol: Servei caigut en servidor

  1. Comprovar l'estat del servei (systemctl status nginx / Get-Service). Llegir el missatge d'error.

  2. Revisar els logs del servei i del sistema per identificar la causa (journalctl -u nginx -b).

  3. Intentar reiniciar el servei (systemctl restart nginx). Si falla, revisar la configuració.

  4. Si la configuració ha canviat recentment: revertir el canvi (git revert si hi ha control de versions) o restaurar la config de backup.

  5. Verificar que el servei funciona correctament i comunicar la resolució als usuaris afectats.

Els 9 continguts del RA8 i les seves aplicacions pràctiques com a tècnics ASIX implantant solucions client/servidor professionals.

📝

Documentació (T1)

Tipus (diagrames, CMDB, runbooks, changelogs). Man pages, RFC, KB. Eines (Confluence, draw.io, Git). Scripts de documentació automàtica. doc-sistema.sh, Get-ComputerInfo.

📖

Manuals (T2)

Estructura: objectiu, prerequisits, procediment, verificació, troubleshooting, rollback. Bones pràctiques: veu activa, blocs de codi, marcar riscos, variables. Provar el manual des de zero.

Llicències (T3)

CAL User/Device/RDS/External. Windows Server per core (Standard/Datacenter). OEM no transferible. Volume Licensing EA/CSP. SPLA per a MSPs. Ubuntu Pro vs RHEL. slmgr /dlv.

🤖

Desatesa (T4–T5)

autounattend.xml (7 passes). cloud-init (user-data YAML): packages, runcmd, users, write_files. Preseed (Debian), Kickstart (RHEL), Autoinstall (Ubuntu 20.04+). multipass launch --cloud-init.

🔄

Actualitzacions (T6)

WSUS: sincronització, aprovació, reporting, GPO per clients. apt-cacher-ng (proxy caché) vs apt-mirror (mirall complet). Estalvi d'ample de banda. wsusutil.exe, apt.conf.d/01proxy.

ITIL v4: detecció → registre → classificació → diagnosi → resolució → tancament. Prioritats P1-P4 i SLA. GLPI (instal·lació LAMP + tiquet complet). Protocols per compte bloquejada, disc ple, servei caigut.
