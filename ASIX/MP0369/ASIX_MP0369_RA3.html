Sistemes de fitxers — Teoria i gestió

NTFS, ext4, Btrfs, XFS, ZFS. Estructura interna, permisos, ACLs i eines de gestió.

Fonaments dels sistemes de fitxers

Què és un sistema de fitxers?

Un sistema de fitxers (filesystem) és l'estructura de dades que el SO utilitza per organitzar i emmagatzemar fitxers a un dispositiu d'emmagatzematge. Defineix com es guarden els fitxers, els metadades associats (permisos, dates, propietari) i l'estructura de directoris. Sense sistema de fitxers, un disc és simplement una seqüència de blocs buits.

🧱

Conceptes fonamentals

  • 📦 Bloc (clúster): unitat mínima d'assignació d'espai. Mida típica: 4 KB. Un fitxer d'1 byte ocupa 4 KB al disc.
  • 🪪 Inode (Linux): estructura que conté tots els metadades d'un fitxer (permisos, propietari, timestamps, adreces de blocs). NO conté el nom del fitxer.
  • 📓 MFT (Windows NTFS): Master File Table — equivalent Windows als inodes. Cada fitxer té al menys una entrada a la MFT.
  • 📋 Journal/Log: registre de transaccions que permet recuperar la consistència del FS després d'un tall de llum.
🪟

NTFS — Windows

New Technology File System. FS principal de Windows NT i successors.

  • ✅ Permisos granulars (ACLs) per fitxer i carpeta
  • ✅ Journaling — integritat en cas de tall
  • ✅ Compressió i xifrat (EFS) per fitxer
  • ✅ Flux d'alternes de dades (ADS)
  • ✅ Volum màxim: 256 TB. Fitxer màxim: 16 TB
  • ⚠️ Propietari Microsoft — Linux accedeix en lectura/escriptura amb ntfs-3g
🐧

ext4 — Linux

Fourth Extended Filesystem. FS per defecte a la majoria de distribucions Linux.

  • ✅ Journaling (3 modes: writeback, ordered, data)
  • ✅ Extents — assignació contigua eficient
  • ✅ Volum màxim: 1 EB. Fitxer màxim: 16 TB
  • ✅ Fins a 4 bilions d'inodes
  • ⚠️ No suporta snapshots natius ni RAID integrat

Permisos i ACLs

Model de permisos: NTFS (Windows) vs POSIX (Linux)

Els permisos de fitxers defineixen qui pot llegir, escriure o executar cada recurs. Windows i Linux usen models completament diferents, tot i compartir objectius similars de control d'accés.

🐧

Permisos POSIX — Linux

Tres entitats: propietari (u), grup (g), altres (o). Tres accions: r(lectura=4), w(escriptura=2), x(execució=1).

Permisos Linux — exemples complets 🐧 Ubuntu
$ ls -la /srv/techcorp/
-rwxr-x--- 1 admin it_admins 4096 fitxer.sh
# r=4 w=2 x=1 → rwx=7 r-x=5 ---=0 → 750
# admin pot tot; it_admins pot llegir+executar; altres: res
# Canviar permisos (forma octal)
$ chmod 750 fitxer.sh
# Canviar propietari i grup
$ chown admin:it_admins fitxer.sh
# ACLs esteses (getfacl/setfacl)
$ setfacl -m u:jdoe:r-- fitxer.sh
$ getfacl fitxer.sh
# file: fitxer.sh
user::rwx
user:jdoe:r-- ← ACL per a jdoe
group::r-x
other::---
# Bits especials: SUID, SGID, Sticky bit
$ chmod u+s /usr/bin/programa # SUID: executa com a propietari
$ chmod +t /tmp # Sticky: sols el propietari pot esborrar
🪟

ACLs NTFS — Windows

Model d'accés basat en ACE (Access Control Entry) agrupades en ACL (Access Control List). Molt més granular que POSIX.

Permisos NTFS — icacls PowerShell 🪟 Windows
# Veure permisos NTFS d'una carpeta
PS> icacls C:\ASIX\dades
C:\ASIX\dades BUILTIN\Administrators:(OI)(CI)(F)
BUILTIN\Users:(OI)(CI)(RX)
# F=Full, RX=Read+Execute, R=Read, W=Write
# (OI)=Object Inherit, (CI)=Container Inherit
# Afegir permís de lectura per a jdoe
PS> icacls C:\ASIX\dades /grant jdoe:(OI)(CI)(R)
# Negar permís explícit (té prioritat)
PS> icacls C:\ASIX\secret /deny jdoe:(OI)(CI)(F)
# Eliminar tots els permisos d'un usuari
PS> icacls C:\ASIX\dades /remove jdoe
# Copiar permisos d'un fitxer a un altre
PS> icacls C:\ASIX\dades /save permisos.txt
PS> icacls C:\ASIX\nous\ /restore permisos.txt

Gestió d'enllaços i cerca d'informació

Hard links, soft links (symlinks), identificació de programari i informació del sistema.

Tipus d'enllaços

Enllaços durs i simbòlics: diferències clau

Un enllaç és una referència a un fitxer o directori. Entendre la diferència entre hard links i symlinks és fonamental per a l'administració de sistemes, especialment per a la gestió d'espai i els scripts d'automatització.

🔗

Hard Link (Enllaç dur)

Un hard link és una entrada de directori addicional que apunta al mateix inode. El fitxer no s'esborra fins que el recompte de hard links arriba a 0.

  • ✅ No hi ha distinció entre "fitxer original" i "hard link"
  • ✅ Si s'esborra un nom, el fitxer segueix existint
  • ❌ No pot creuar sistemes de fitxers (han d'estar al mateix FS)
  • ❌ No funciona per a directoris (excepció: . i ..)
Hard links🔀 Ambdós
# Linux: ln (sense -s)
$ ln original.txt hardlink.txt
$ ls -li # Mateix inode!
1234567 -rw-r--r-- 2 user group original.txt
1234567 -rw-r--r-- 2 user group hardlink.txt
# Windows: mklink /H
> mklink /H hardlink.txt original.txt
🔀

Soft Link / Symlink (Enllaç simbòlic)

Un symlink és un fitxer especial que conté la ruta d'un altre fitxer o directori. Similar a una drecera de Windows, però a nivell de SO.

  • ✅ Pot apuntar a directoris i a fitxers en altre FS
  • ✅ Té el seu propi inode (diferent del destí)
  • ❌ Si s'esborra el fitxer original, el symlink queda trencat (dangling)
  • ❌ Lleuger overhead en accés (resolució de ruta)
Symlinks🔀 Ambdós
# Linux: ln -s (ruta_destí, nom_link)
$ ln -s /srv/dades/versio3 /srv/actual
$ ls -la /srv/actual
lrwxrwxrwx 1 root root 20 actual -> /srv/dades/versio3
$ readlink -f /srv/actual
/srv/dades/versio3
# Windows: mklink (directori: /D, junction: /J)
> mklink link.txt C:\dades\fitxer.txt
> mklink /D DirLink C:\dades\versio3

Discos, particions, volums i muntatge

MBR vs GPT, particions primàries/esteses/lògiques, LVM, muntatge, fstab i desfragmentació.

Esquemes de particionament

MBR vs GPT — la diferència crítica

L'esquema de particionament defineix com s'organitzen les particions al disc. La tria entre MBR i GPT té implicacions en la mida màxima del disc, el nombre de particions, la compatibilitat amb UEFI i la resiliència davant errors.

🕰️

MBR — Master Boot Record (Legacy)

  • 📏 Límit de disc: 2 TB màxim. Discos més grans no funcionen correctament.
  • 🗃️ Particions: Màxim 4 primàries, o 3 primàries + 1 estesa (amb particions lògiques il·limitades).
  • 🔒 Arrencada: BIOS (Legacy). Incompatible amb Secure Boot modern.
  • ⚠️ Resiliència: Una còpia del MBR al primer sector. Si es corromp, el disc no arrenca.
🆕

GPT — GUID Partition Table (Modern)

  • 📏 Límit de disc: 9,4 ZB (pràcticament il·limitat).
  • 🗃️ Particions: Fins a 128 particions primàries (cap partició "estesa").
  • 🔒 Arrencada: UEFI. Suporta Secure Boot. Compatible amb Windows 11 (obligatori).
  • ✅ Resiliència: Còpies de la taula de particions al inici I al final del disc. Molt més robust.
LVM — Linux Volume Manager

LVM: gestió lògica de volums a Linux

LVM (Logical Volume Manager) afegeix una capa d'abstracció entre el hardware físic i els sistemes de fitxers. Permet redimensionar volums en calent, crear snapshots i agregar múltiples discos físics en un únic volum lògic sense repartir.

LV (Logical Volumes): /dev/vg_data/lv_home (50G) + /dev/vg_data/lv_srv (100G)
Volums lògics — el que veu el SO. Redimensionables en calent
VG (Volume Group): vg_data (150G total = sdb 100G + sdc 50G)
Grup de volums — pool d'espai lliure agregat de múltiples discos
PV (Physical Volumes): /dev/sdb1 (100G) + /dev/sdc1 (50G)
Volums físics — particions o discos inicialitzats per LVM
Gestió de discos i LVM — Ubuntu i Windows🔀 Ambdós
═══ UBUNTU — fdisk, parted, LVM ═══
$ lsblk -f # Llistar blocs, FS i muntatge
$ sudo fdisk -l /dev/sdb # Informació de particions del disc
$ sudo parted /dev/sdb print # Parted: suporta GPT
# Crear partició GPT amb parted
$ sudo parted /dev/sdb mklabel gpt

Muntatge i /etc/fstab

Muntatge de dispositius i automatització amb /etc/fstab

/etc/fstab — format i exemples🐧 Ubuntu
# Camps: dispositiu punt_muntatge tipus opcions dump pass
UUID=abc123 / ext4 errors=remount-ro 0 1
UUID=def456 /home ext4 defaults 0 2
/dev/sdb1 /mnt/dades ext4 defaults,noatime 0 2
//srv/share /mnt/nas cifs credentials=/etc/samba.cred,uid=1000 0 0
tmpfs /tmp tmpfs mode=1777,size=2G 0 0
# Obtenir UUID d'un dispositiu
$ blkid /dev/sdb1
/dev/sdb1: UUID="abc123..." TYPE="ext4"
# Muntar tot el fstab (verifica errors)
$ sudo mount -a
$ sudo systemd-analyze verify /etc/fstab
# Muntatge temporal
$ sudo mount -t ext4 /dev/sdb1 /mnt/temp
$ sudo umount /mnt/temp
⚠️
Usa sempre UUID al fstab, mai /dev/sdX

El nom /dev/sda, /dev/sdb pot canviar entre reinicis (ordre de detecció del kernel). L'UUID és permanent i únic per cada partició. Un fstab incorrecte pot impedir l'arrencada del sistema!

💡
Opcions de muntatge importants:
  • noatime — No actualitza temps d'accés. Millora rendiment en SSD.
  • ro — Muntatge en mode lectura. Per a discos de recuperació.
  • noexec — Evita executar binaris. Seguretat extra a /tmp.
  • nosuid — Ignora bits SUID/SGID. Seguretat extra.

RAID — Redundant Array of Independent Disks

RAID0 (rendiment), RAID1 (mirall), RAID5 (paritat), RAID10. Implementació per programari.

Conceptes RAID

Tipus de RAID: rendiment vs tolerància a errors

RAID és una tecnologia que combina múltiples discos físics per obtenir avantatges de rendiment, capacitat o tolerància a errors (o una combinació). El RAID no substitueix les còpies de seguretat — protegeix contra falles de disc, però no contra errors humans, virus o desastres.

Còpies de seguretat — Teoria i implementació

Tipus de backup, estratègies (3-2-1), eines professionals, plans i programació.

Tipus de còpies de seguretat

Completa, incremental i diferencial

Triar el tipus de còpia de seguretat adequat és fonamental per equilibrar el temps de còpia, l'espai consumit i el temps de recuperació. No existeix un tipus "millor" universal — la decisió depèn dels requisits del negoci.

💯

Còpia completa (Full)

Copia absolutament tots els fitxers cada vegada. El bit d'arxiu es neteja.

  • ✅ Recuperació: 1 sol conjunt de cintes/arxius
  • ✅ Màxima velocitat de restauració
  • ❌ Molt de temps i espai per còpia
  • 📅 Normalment: setmanal o mensual
➕

Còpia incremental

Copia només els fitxers canviats des de l'última còpia (completa o incremental). El bit d'arxiu es neteja.

  • ✅ Molt ràpida i ocupa poc espai
  • ❌ Recuperació: necessita Full + TOTES les incrementals en ordre
  • ❌ Restauració complexa i lenta
📊

Còpia diferencial

Copia tots els fitxers canviats des de l'última còpia COMPLETA. El bit d'arxiu NO es neteja.

  • ✅ Recuperació: Full + l'última diferencial
  • ⚖️ Velocitat i espai: entre full i incremental
  • 📅 Creix cada dia fins a la pròxima Full
Plans de còpies de seguretat

Estratègia de backup: regla 3-2-1 i plans combinats

La regla 3-2-1 és el estàndard de facto per a la protecció de dades: 3 còpies dels dades, en 2 suports físics diferents, amb 1 còpia fora del lloc (off-site o núvol).

Exemple de pla setmanal combinat

🗓️ Diumenge 02:00 — Còpia Completa

Tots els sistemes. Durada: 4h. S'emmagatzema al NAS local + replicació al núvol (S3). Reté les últimes 4 còpies completes.

📅 Dl–Ds 03:00 — Còpia Incremental (o Diferencial)

Només fitxers nous o modificats. Durada: 30-60 min. Al NAS local. Retransmissió al núvol comprimida i xifrada (AES-256).

🔄 Mensual — Arxivament a llarg termini

Còpia completa a cinta LTO o a emmagatzematge fred (S3 Glacier). Reté 12 mesos. Per a compliment normatiu (RGPD).

🧪 Trimestral — Test de recuperació

Restauració completa en entorn de test. Documenta RPO i RTO reals. Un backup no provat no és un backup!

📐
RPO i RTO — Mètriques crítiques:
  • RPO (Recovery Point Objective): Quanta informació es pot permetre perdre. Un RPO d'1 dia significa que en cas de desastre perdrem com a màxim les dades de les últimes 24h.
  • RTO (Recovery Time Objective): Quant de temps pot estar el sistema aturat. Un RTO de 4h significa que en 4h s'ha de tenir el sistema operatiu.
  • El backup ha d'estar dissenyat per complir el RPO/RTO definit pel negoci.
⚠️
Errors comuns de backup que cal evitar:
  • Fer backup i restaurar al mateix disc (RAID1 no protegeix de borrar fitxers)
  • No provar mai la restauració (un backup invàlid és inútil)
  • No xifrar les còpies fora del lloc
  • No automatitzar (el backup manual s'oblida)
  • Confondre RAID amb backup

Quotes de disc — Nivells i advertències

Gestió d'espai per usuari i grup. Soft quota, hard quota i grace period. NTFS i Linux.

Conceptes de quotes

Quotes de disc: controla l'espai per usuari

Les quotes de disc permeten a l'administrador limitar l'espai que cada usuari (o grup) pot consumir a un sistema de fitxers. Sense quotes, un sol usuari podria omplir tot el disc i deixar el sistema sense espai per als altres.

📊

Soft Quota (Límit suau)

Límit que es pot superar temporalment. Quan es sobrepassa, comença el compte del grace period.

Usuari: jdoeSoft: 4GB
Ús actual: 3,2 GB de 4 GB (soft) / 5 GB (hard). ⚠️ Grace period actiu: 6 dies restants.
🚫

Hard Quota (Límit dur)

Límit absolut. Quan s'arriba, el sistema denega qualsevol escriptura addicional. No hi ha excepcions.

Usuari: jdoeHard: 5GB
❌ Quota hard assolida. Escriptures denegades. Qualsevol intent de guardar arxius fallarà.
⏱️

Grace Period

Temps que l'usuari té per reduir el seu ús per sota de la soft quota un cop superada.

  • ⏰ Valor típic: 7 dies (configurable)
  • 📧 Genera avisos automàtics per correu
  • 🔴 En exhaurir-se, la soft quota es comporta com hard quota

Resum del RA3

El que heu après

Repàs dels conceptes clau del RA3 i la seva aplicació professional com a tècnics ASIX.

🗂️

Sistemes de Fitxers

FS: NTFS, ext4, Btrfs, XFS, ZFS. Inodes, blocs, journaling. Permisos: POSIX (rwx, SUID, SGID, sticky) + ACLs esteses. Eines: chmod, chown, setfacl, icacls. LVM: PV→VG→LV, ampliació en calent.

💾

Discos i RAID

Particions: MBR (≤2TB, 4 primàries) vs GPT (≤9,4ZB, 128 particions). RAID: 0 (rendiment), 1 (mirall), 5 (paritat, mínim 3 discos), 6 (2 fallades), 10. mdadm (Linux), Storage Spaces (Windows).

🔒

Backup

Tipus: completa, incremental (ràpida, restauració complexa), diferencial (equilibri). 3-2-1: 3 còpies, 2 suports, 1 off-site. Eines: rsync, tar+GPG, wbadmin, Timeshift. Clau: testa la recuperació!

🔧

Recuperació

Linux: Live USB + chroot + grub-install. fsck, testdisk, PhotoRec. LVM snapshots. Windows: WinRE, bootrec, bcdboot, SFC, DISM. Quotes: soft+grace period, hard (absolut). edquota, repquota.
